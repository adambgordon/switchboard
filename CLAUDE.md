# Switchboard — project guide

A macOS Electron app — light or dark theme — that browses Claude Code conversations: read-only transcript preview, resume/new into a real terminal (node-pty + xterm.js), a live "Formatted" view, and pinning. The layout is two columns — a single unified left pane (Pinned / Live / Recent conversations) and a main pane. **All conversation data is derived from the JSONL files Claude Code writes under `~/.claude/projects/`** — Switchboard never owns it. The one exception is **rename**, which appends Claude Code's *own* `custom-title` line (the `/rename` record) — so even that isn't a private store; the title is real and survives into `claude --resume`. See the **Conversation rename + info modal** pointer.

Stack: Electron 42 (pinned as `latest`, so it tracks upstream) · electron-vite · React 18 + Vite · TypeScript (strict) · plain CSS · node-pty · chokidar · @xterm/xterm · react-markdown · fuse.js. No Tailwind, no state library (plain hooks + localStorage).

## Set up Switchboard (first run)

When the user asks to **"Set up Switchboard"** (typically from a fresh clone — this is the phrase the public README's "Set up" section points them at), run the one-shot setup and then open the built app:

```bash
npm run setup
open dist/mac-arm64/Switchboard.app
```

`npm run setup` (= `scripts/setup.mjs`) installs dependencies, rebuilds `node-pty` for Electron's ABI, and packages `dist/mac-arm64/Switchboard.app` — but it does **not** launch the app, so always follow it with `open`. If it fails on a missing prerequisite (Xcode Command Line Tools, or Node 26 — see `.nvmrc`), walk the user through installing it, then re-run.

## Update Switchboard

When the user asks to **"Update Switchboard"** (the phrase the public README's "Updates" section points them at), pull the latest and re-run the one-shot build, then reopen:

```bash
git pull
npm run setup
open dist/mac-arm64/Switchboard.app
```

`npm run setup` rebuilds `dist/mac-arm64/Switchboard.app` from the pulled source (deps + `node-pty` + package). Tell the user to **quit (⌘Q) any running instance first** — `open` only focuses an already-running app, so a stale instance won't pick up the new build otherwise. If `git pull` reports local changes/conflicts or the build fails on a prerequisite, surface that rather than forcing past it.

## Build, validate, run

The building blocks (run individually as needed):
1. `npm run typecheck` — `tsc` over BOTH projects (main = node, renderer = web). Must be clean.
2. `npm run build` — `electron-vite build` → `out/`. Must succeed. **Does NOT update the `.app`** — `out/` is what `npm run dev` and the smoke check consume, not the packaged bundle.
3. **Boot smoke:** `SWITCHBOARD_SMOKE=1 node_modules/.bin/electron .` — prints `SMOKE PASS | pty:true | window:loaded`. The only headless check that node-pty loads/spawns under Electron's ABI and the renderer loads. `window:none` is a sampling race (re-run); only `did-fail-load` / `preload-error` lines indicate a real failure.
4. `npm test` — vitest (parser / indexer / customTitle). The renderer is not unit-tested.
5. `npm run package` — `electron-vite build` + `electron-builder --dir`, rebuilding the installed `dist/mac-arm64/Switchboard.app`. `npm run build` alone does NOT do this. Quit (⌘Q) and reopen the app to load it.

For iterating, `npm run dev` (hot reload) is the inner loop. The renderer (visual feel, live terminal, drag/scroll) can only be truly verified by a human running `npm run dev` or the freshly-packaged `.app` — typecheck/build/smoke don't catch React-runtime or visual issues. Say so rather than claiming the UI works.

Set `SWITCHBOARD_DEV_LABEL=<name>` before `npm run dev` to tag the window title **and** the title bar (a mono pill left of the gear) — so several dev instances can run side by side and stay tellable apart (there's no single-instance lock; Vite auto-increments the port). Unset in normal and packaged runs, where the title stays a plain "Switchboard".

## Gotchas (these have bitten before)

- **node-pty rebuild:** use `npm run rebuild` (= `node scripts/rebuild-native.mjs`). The `@electron/rebuild` CLI crashes under Node 26 (yargs/ESM), so we call its programmatic API directly.
- **⌘R refreshes the terminal — it does NOT reload (the app installs a custom menu).** Electron's default menu binds ⌘R / ⇧⌘R to the `reload` / `forceReload` roles, which is destructive here: a renderer reload destroys every xterm buffer, and the main process keeps **no** PTY output to replay (`PtyManager` only streams live bytes; the renderer-side `ptyStream` backlog is wiped on reload) — so the remounted `TerminalView`s attach to an **empty** backlog and an idle `claude` emits nothing ⇒ every live terminal goes **blank**, recoverable only by a *real dimension change* (which is why a manual zoom was the one thing that "fixed" it). `src/main/menu.ts` (`installAppMenu`, called from `index.ts` `whenReady`) replaces the default menu with mostly **role-based submenus** (`appMenu` / `editMenu` / `windowMenu`, plus a custom **File** = Close Window + Quit — so the standard menus and shortcuts, copy/paste/quit/close/zoom/window, are untouched; **don't drop the File menu** — its absence reads as a broken app. Quit sits in both the app menu (macOS convention) and File; the shared ⌘Q is harmless since both quit) and a custom **View** that drops the reload roles and rebinds **⌘R / ⇧⌘R → a zoom-wiggle refresh** (`refreshFocused`, surfaced as the **Refresh** menu item): nudging `webContents.setZoomLevel` by a fraction of a level (`REFRESH_ZOOM_NUDGE`, ≈0.5) and restoring it relayouts the renderer → the terminal host's `ResizeObserver` → `fitAndResize` → a real SIGWINCH (claude full redraw) **and** a full-page recomposite (which clears a *stale WebGL frame* — a bare terminal resize can't, see the stale-composite note below; this is why the refresh is a genuine zoom and not a targeted `term.refresh()`). The nudge is kept **small to minimize the flicker** but must stay large enough to cross a terminal cell-row boundary at any window size ≥ the app minimum — too small and it rounds to the same cols/rows ⇒ no SIGWINCH ⇒ no content redraw; restored after ~60ms (an out-and-back within one frame coalesces to a no-op). **Don't reintroduce the `reload` / `forceReload` roles**, and if you add menu items keep building from the role-based template so the standard shortcuts survive.
- **Packaging:** if you have no code-signing identity, package with `CSC_IDENTITY_AUTO_DISCOVERY=false`. macOS may Gatekeeper-prompt on first open of a fresh build.
- **localStorage is per-origin:** dev (`http://localhost`) and the packaged `.app` (`file://`) don't share pins/layout. Expected, not a bug.
- **macOS Option key:** for `⌘⌥` shortcuts use `e.code` (e.g. `'KeyB'`), not `e.key` (Option+B yields `∫`).
- **Terminal image input rides on bracketed paste.** claude attaches an image on a *paste* — it reads the system clipboard, or parses an image path inside the pasted text — never on plain typed text. So `TerminalView` delivers via the bracketed-paste sequence (`ESC[200~ … ESC[201~`): **Ctrl+V** sends an *empty* one (claude then reads the clipboard image → `[Image #N]`); **drag-drop** wraps the dropped, shell-escaped path(s) in one. The Cmd+V/Ctrl+V roles are **inverted vs. a native terminal**: under xterm, **Cmd+V** is the real (browser) `paste` event while **Ctrl+V** would only send `\x16`. So we own **Ctrl+V** via `attachCustomKeyEventHandler`, and intercept **Cmd+V** in *capture phase* to keep it text-only (an image-only clipboard is suppressed, so claude can't grab it). macOS-only (Cmd+V fires a DOM `paste` event; Ctrl+V doesn't).
- **Terminal renders via the WebGL addon (canvas is the fallback), never the DOM renderer.** `TerminalView` loads `@xterm/addon-webgl` *after* `term.open()` (required ordering); on GPU-context loss (`onContextLoss`) — or if WebGL can't initialize — it swaps to `@xterm/addon-canvas`. WebGL is the faster, higher-fidelity renderer (it draws every glyph each frame *without* clipping it to the cell, which canvas does — that clipping was the source of the box-glyph/whitespace artifacts) and is where xterm.js + VS Code are consolidating. Both honor `customGlyphs` (default on), so box-drawing/block glyphs paint as continuous vector lines. The fallback is **canvas, not the default DOM renderer**: DOM pulls box-glyphs from the font, which breaks `claude`'s table/box borders at our non-integer `fontSize` (11.5) + `lineHeight` (1.3). Don't switch the *primary* off WebGL or the *fallback* to DOM without eyeballing how tables render. (Text stays IBM Plex Mono either way.)
- **Terminal `lineHeight` is kept near-integer for the canvas fallback.** `11.5 × 1.3 ≈ 15px` makes the cell height ~integer. The WebGL primary doesn't need this, but the **canvas fallback** strands blank rows during rapid live redraws at a fractional cell (claude's tool-call output) — intermittent vertical whitespace gaps that only a repaint (scroll, or ⌘ +/- zoom) clears, never selection (selection is a separate render layer). It was `1.32` (15.18px); WebGL fixed those gaps at the root (no per-cell clipping / repaint desync), but keep `1.3` so the fallback path stays clean. Don't raise `lineHeight` to a value that makes the cell fractional again.
- **Terminal Unicode width is pinned to v11.** `TerminalView` loads `@xterm/addon-unicode11` and sets `term.unicode.activeVersion = '11'` right after `term.open()` (before the first PTY write). xterm's built-in default is the Unicode **V6** width table, which measures emoji-presentation chars (✅ `U+2705`, ❌ `U+274C`, …) as **1 cell** — but `claude` lays out tables / box-art assuming the modern **2-cell** emoji width (as every real terminal does), so the 1-cell mismatch shifted every cell after an emoji and knocked the closing `│` border out of alignment (and the half-cell glyph overdraw flickered under partial canvas repaints on scroll/select). V11 widens those codepoints to 2, matching claude. Requires `allowProposedApi: true` (the `term.unicode` interface is proposed API). Don't remove the addon or fall back to V6 — and don't confuse this with grapheme clustering, a separate, costlier feature we deliberately don't enable (V11 is just a corrected width table; the per-char hot path is unchanged from V6).
- **Terminal cursor is hidden — Switchboard defers to Claude's own caret.** `TerminalView` sets `theme.cursor` transparent (`rgba(0,0,0,0)`) with `cursorBlink: false`. `claude` (like most full-screen TUIs) draws and styles its OWN caret — a reverse-video block — so xterm's hardware cursor would just paint a redundant *second* cursor on the same cell (it showed as a cobalt bar beside Claude's block). Transparent leaves Claude's block as the only caret. `cursorStyle: 'bar'` is kept solely so the invisible cursor never recolors the glyph under it (a `block` cursor would), and this subsumes the old `cursorInactiveStyle: 'none'` too (a transparent cursor is invisible focused or not, so arrow-preview shows no stray box). Trade-off: no visible cursor at a bare shell prompt — fine, since this terminal exists to host a live `claude`. Cursor *style* is therefore not a Preference (deliberately not among the App page's settings) — it's Claude's to draw.
- **A hidden terminal's renderer is paused, so it's force-repainted on show.** xterm's `RenderService` installs an `IntersectionObserver` on the screen element and sets `_isPaused` while it isn't intersecting the viewport — which `display:none` triggers — after which `refreshRows` **drops** every paint request. `TerminalDeck` hides inactive live sessions with `display:none`, so a session that animates while hidden (claude's mid-thought spinner) freezes on its last painted frame when you navigate to it, and stays frozen until a redraw is forced (a manual scroll was the only cue). `TerminalView`'s becomes-visible effect fixes it: it **clears that paused flag directly** (`term._core._renderService._isPaused = false`, reached via private API and guarded with optional chaining) before the fit + `term.refresh(0, rows-1)`. The earlier fix — refreshing on two `requestAnimationFrame`s and hoping the observer had un-paused — was unreliable, because Chromium does **not** consistently re-fire the `IntersectionObserver` on a pure `display:none → block` toggle (it recomputes intersection on scroll/layout), so the refresh got dropped while still paused (symptom: a frozen spinner that only a manual scroll woke). Clearing the flag ourselves guarantees the repaint; the observer later sets it correctly (visible ⇒ false), a no-op. One-time per navigation (no per-frame cost). Don't drop it — and don't "fix" it by taking the deck off `display:none`, which would keep every hidden live terminal repainting (a CPU regression against the LRU/pause bounds).
- **Never fit/resize a hidden (`0×0`) terminal — it pushes a bogus tiny size that strands blank rows.** When a deck item goes `display:none` (you switch to another conversation), its host measures `0×0` *and* its `ResizeObserver` fires — but `@xterm/addon-fit` does **not** bail on a zero-size host: it reads the host's `height`/`width:100%` *computed* style (`parseInt("100%")` → 100px) and proposes a ~`13×5` grid. Pushing that to the PTY makes `claude` reflow its whole TUI into 13×5 and then back to full size when you return, and the reflow strands **real blank rows** in the buffer (the "whitespace after tool calls" gap — same symptom as the 80×30 boot bug, but a *different* cause: a live resize, not boot replay). `TerminalView` funnels every resize through one `fitAndResize` that **skips entirely when `host.clientWidth`/`clientHeight === 0`** and **suppresses no-op resizes** (sends only when `cols`/`rows` actually changed), so switching conversations/views sends *zero* resizes when the size is unchanged. Don't reintroduce an unconditional `window.api.resize(...)` after `fit.fit()`, and don't drop the `0×0` guard — the `ResizeObserver` firing on hide is the trigger. (The guard also subsumes the boot case: a hidden mount no longer sends xterm's 80×24 default, so it can't satisfy the boot-size gate at the wrong size.)
- **The initial terminal fit runs before the PTY backlog, on purpose.** `TerminalView` calls `fitAndResize()` once right after `term.open()` + addons and *before* `attachPty`, so the expensive WebGL resize hits an empty terminal instead of a screenful mid-replay — it removed a ~170ms main-thread stall on resume (trace-verified). The rAF + `ResizeObserver` after it are the fallback (host not yet measurable) and the later-resize path; don't move the first fit back behind the flood. The residual *choppy scroll for the first seconds after resume* is GPU/compositor frame-pacing — not main-thread, not cheaply fixable — so don't re-chase it as a main-thread problem.
- **Fonts are warmed before first paint.** `main.tsx` gates `root.render` on `document.fonts.load(...)` for every bundled weight (Hanken 400/500/600/700/800, IBM Plex Mono 400/500/600), raced against a 1500 ms ceiling. `@fontsource` ships each weight as a separate `font-display: swap` file fetched lazily on first use, so a bold run could paint in a fallback face and then **not** repaint when the real face loaded — Chromium leaves the already-painted glyphs stale until a reflow (selecting or zooming the text forced it). Decoding up front closes that window: text paints once, in the right face (~20 ms on local files, behind the `show:false` window). Don't drop the gate or the intermittent bold-font flash returns — and note `document.fonts.ready` *alone* is a no-op here (nothing is pending before render, so the explicit `.load()` calls are what kick off the decode).
- **Don't clamp `white-space: pre-wrap` text with `-webkit-line-clamp`.** The Formatted view caps a long tool result at **12 rendered lines** via a content-box `max-height` (= 12 line-heights, `1.55em` each) on a **padding-less inner clip** element with `overflow:hidden`; the gap below the cut is the panel's own bottom padding. `-webkit-line-clamp` was tried (twice) and fails on `pre-wrap`: it cuts **mid-line** AND makes `scrollHeight === clientHeight`, which broke the overflow check that gates "Show more". That check is now a `ResizeObserver` on the text (a one-shot read raced layout + font-load and dropped the button).

## Design invariants (don't break these)

- **Two accent colors, each with one job.** Cobalt (`--live`, `#1F5AE6`) = a session is *alive*; the only *action* that may carry it is the one that brings a conversation to life — the right-click **Resume** (cobalt-on-hover, the live counterpart to Stop's red). A warm red (`--danger`, `#d8392c`) = *negative / destructive* (End/Stop controls, error states). Everything else — pins, selection, metadata — stays ink/graphite. **Selection is a single tier** — the main pane always owns the keyboard, so there's no list-focus state: the selected row carries its colour as a thin **1px ring**, **cobalt** for a *live* one and neutral grey/ink for the rest (cobalt stays liveness-only, so a cobalt ring = the selected conversation is live), with **no drop shadow or lift**. Mouse-**hover** keeps its own card lift — the theme-aware `--shadow-sel-hover` token: in **light** a dark drop shadow, in **dark** a neutral **light glow** (a *black* shadow is invisible on the near-black rail) — kept narrow + downward-biased so it never hard-cuts at `.sb-rail-body`'s `overflow` clip or the scrollbar gutter; the selected/hovered row is `z-index:1` so that lift paints over neighbouring live cards. (The old focus-tiered *loud* selection shadow and its `--shadow-sel` token were removed when list-focus went away.)
- **Row controls live in the right gutter.** In the left pane's rows, the liveness indicator and the pin toggle share one right-hand slot: at rest a live row shows the cobalt liveness dot (everything else is blank), whose form encodes the conversation's turn-state — **breathing** = a turn is in progress, **a cobalt ripple** (rings pulsing outward from a briefly-swelling dot) = Claude is blocked on your reply (an AskUserQuestion / ExitPlanMode, unread), **solid** = the turn finished and you haven't viewed it since (unread), **hollow ring** = finished and already seen (up to date); the solid/unread state can also be set **by hand** (`⇧⌘U`, **Option+click** on a live row, or a row's right-click menu — see the Liveness model below); hovering a **non-live** row reveals the pin/unpin toggle (filled = pinned, outline = not); on a **live** row the pin appears only when you hover the **gutter** itself (the dot's slot), which fades the dot out behind it — so skimming a live row's body never disturbs its liveness dot. The pin is **never shown at rest, even when pinned** — the **Pinned** section already says it's pinned. Rows no longer carry an inline Stop; a live session is ended by the **Stop** button in the pane header, or a live row's right-click → **Stop session**. (The pane header keeps its own pin on the **left** of the action cluster; there, the **Terminal** view-toggle carries a **static solid cobalt dot** — it marks that the session is live but no longer encodes turn-state; the breathing/ripple liveness animation lives only on the left-pane rows.) **Pinned and Live** rows can each be **click-dragged** vertically to reorder within their own section (`useRowReorder`, one instance per section partitioned by CSS selector — `.sb-row.pinned` vs `.sb-row.live:not(.pinned)`, so a pinned-live row is owned by the Pinned instance; the moving card is a `position:fixed` clone on `<body>` so it floats over the head instead of being clipped by the rail body's overflow; its vertical travel is **clamped to a partial overshoot** past its section (≈ half a row — the card's center can't leave the section), not free; siblings slide live to open the drop gap, fixed section height; on drop the clone eases into its slot *while* dropping out of the lifted look at the same time, so release reads as one motion, not a slide-then-snap-flat); reorder is always on.
- **The pane orders by stable keys, never live activity.** Rows hold position by pin order (**Pinned**) and a **manual order** (**Live** — every newly-live session, new or resumed, lands on top, and you drag to reorder; held in `useLiveOrder`, ephemeral) — never `lastActivity` — so a session emitting output never makes a live row jump (regression fixed in `ce407fa`; the manual order now makes this structural — Live no longer auto-sorts by `startedAt` at all, a row holds its slot until you move it). The **Recent** section is the deliberate exception: a recency history of *not-live* conversations, sorted by `mtime`. Pinning a row jumps it to the top of **Pinned** — but that's a user action, not activity. **The whole pane renders through `ConversationRow`** — the unified left pane *is* the left column now (no separate sidebar): card chrome (`.sb-row.card`) marks **live rows only** (a white card = a live session, so a non-live pinned row renders flat like Recent), and Pinned/Live add the live click-to-jump; the pin-on-hover gutter toggle is a shared `.sb-row` primitive, so it (and the liveness dot) work on Recent's flat rows too. The three sections are individually collapsible, and **Recent** is capped (30) with a *Show more* reveal at the bottom. So `ConversationRow` changes flow into every section. When a row *does* change section — pinned, resumed, or unpinned — it **glides** to its new position via a FLIP animation (`useRailFlip`) instead of teleporting; the old `sb-patch-in` entrance keyframe was retired (it re-fired when a row gained `.card` in place, collapsing then re-expanding a resumed pinned row — the "delete + re-add" flicker), so a row going live just eases its card *background* in. `border-color` and `box-shadow` are deliberately **not** transitioned on `.card` rows, so the selection ring (`.selected`, a single thin 1px tier) is **instant** on select *and* deselect — matching the flat rows, which snap the same way; transitioning them made the ring linger ~120ms (an expo-out tail) when you switched to another conversation. (Card hover snaps too, as a consequence — consistent with the rail's flat rows.)
- **Single-family type:** Hanken Grotesk for UI + display (heavy weights carry the display moments); IBM Plex Mono for the terminal + mono metadata. No serif. (One narrow exception: keyboard-shortcut glyphs — `⌘ ⇧ ⌥ ⏎ ↑ ↓` — render in `--font-keys` = the macOS system font, which carries them at consistent size/baseline; Hanken & Plex lack several and fall back per-glyph, which looked small/off-baseline.)
- **Preview ≠ spawn:** selecting a conversation only previews it (read-only, from the JSONL). A live `claude` process is created ONLY by explicit **Resume** or **New**.
- **Formatted view is read-only** (rendered from the JSONL); typing happens in the **Terminal** view. It is also the surface for **find-in-conversation** (`⌘F` when the main pane is focused) — and find now **reveals** collapsed sections (opens a closed tool `<details>`, expands a clamped result) when you navigate to a hidden match; only transcript chrome is excluded.
- **Transcript rendering is one `MessageBlock` per *group*, not per message.** `lib/messageGroups.ts#buildGroups` coalesces consecutive same-source lines under a single header — Claude → Claude → Result → Result renders as one **Claude** then one **Result** (memoized on the transcript so search re-renders keep `MessageBlock`'s memo). **Attribution is by content, not raw role:** the parser tags each `user` line `userKind` — `human` (typed prompt → "You"), `tool_result` (tool output → a **Result**/**Error** header, never "You"), or `interrupted` (the Esc sentinel → a centered muted note); non-conversational `user` lines (slash-command echo, `isMeta`, task-notifications) **and thinking blocks** are dropped at parse time. A single hairline rule separates a **You** (human) turn from the surrounding Claude / Result / Error turns — drawn ONLY at that boundary, never between same-side turns (`.message.has-divider`, computed in `TranscriptView` as `dividerBefore`). The **You** body renders as plain `white-space:pre-wrap` prose (same ink/size as Claude's text, no box, **not** markdown-parsed — so typed line breaks survive). Tool calls/results are **boxless** — the `⚙ Name` head is itself the click-to-expand disclosure (gear only, no chevron, hover-lightens, `user-select:none`) over the sunken JSON/result block — **except an error result, which keeps its red box**. Each header carries a quiet right-aligned timestamp (`format.ts#clockTime`; full date/time on hover via `data-tip`). Code/mono surfaces sit at the **terminal size** (`--text-xs`). **Copy affordances** are hover-revealed icon buttons (shared `CopyButton`, neutral ink, check-flash on click like `copySessionId`): a **turn** copy immediately left of the timestamp — only on **You / Claude** turns that carry prose, copying the **raw markdown source** of the turn's text blocks (`lib/clipboard.ts#turnMarkdown`, pure + unit-tested). The assistant's text is stored AS markdown, so this preserves headers / bold / inline code / fenced code / pipe tables with the author's own blank-line spacing; non-text blocks (tool calls, tool results, images) are skipped, so the ⚙ gear I/O never lands in the copy. (This replaced an earlier DOM-`innerText`-plus-sentinel scheme — the stored source is simpler and exact.) A **corner** copy on every mono block — markdown code (full text), tool-call JSON (`safeStringify(input)`), tool-result text (the full `text` prop, even when clamped), and markdown tables (the **raw markdown source**, sliced from the original via the hast `node`'s position offsets so cell formatting / alignment survive — same as the turn copy; falls back to `lib/clipboard.ts#rowsToMarkdownTable` DOM reconstruction if positions are missing). Buttons carry no text so they're invisible to find; the per-block ones ride a `position:relative` non-scrolling wrapper so they don't scroll with the block.
- Design tokens live in `src/renderer/styles/tokens.css` — reuse the CSS vars; don't hardcode colors.
- **Two themes, one source of truth — a hardcoded color is a dark-mode bug, and EVERY visual change must be authored + verified in BOTH themes.** `tokens.css` defines a **neutral grayscale** light palette in `:root` and a **neutral near-black** dark palette in `:root[data-theme='dark']` (only color + shadow tokens are overridden; type/spacing/radii/motion/layout are shared). Surface tokens: `--paper-sunken` = **chrome** (the rail AND the title bar — `.sb-titlebar` uses `--paper-sunken`, not `--paper`, so they read as one continuous chrome), `--paper-pane` = **main content** (pane / Formatted / terminal), `--paper-raised` = **raised tiles** (live cards, modals, menus), `--paper` = the field/shell behind them. **Dark deliberately INVERTS the elevation** (see the dark-scope comment): the chrome (`--paper-sunken`) is *lighter* than the content canvas (`--paper-pane`), and tiles (`--paper-raised`) lighter still — the reverse of light mode and of the usual dark convention. Every surface paints from a token, so dark mode is "free" for any component that obeys this — **but "free" covers hue, not feel.** A token flips the *color* per theme; it does NOT carry perceptual *strength* across the light↔dark luminance gap, so any change with a visual dimension (shadow alpha, opacity, contrast, glow) must be **eyeballed in BOTH themes**, not just authored in one. The rail's card hover-lift shadow is the cautionary tale, with a twist: a *low-alpha* black drop shadow is **invisible** on the near-black rail (no contrast at the dark end), so the theme-aware `--shadow-sel-hover` token shares one geometry but **boosts the alpha to near-opaque in dark** — a faint black vanishes on the `--paper-sunken` rail, while an opaque one darkens toward #000 and reads (light keeps the soft low-alpha black). The cobalt **Resume** button's hover reuses the same token. "I changed a visual and only looked in light" is unfinished work. The **two-accent invariant holds in both themes**: cobalt = alive, red = destructive (both brightened for dark, soft/wash tints dark-tinted); pins/selection/metadata stay ink. One-off shadows use the `--shadow-color` channel (`rgba(var(--shadow-color), α)`) and overlays use `--scrim`. The Preferences modal mirrors these surfaces (chrome head+nav, content page — see the Preferences pointer). The active theme is a JS-resolved `data-theme` on `<html>` (System/Light/Dark), **never** a bare `@media (prefers-color-scheme)` — so explicit Light/Dark beats the OS. See the **Theming** architecture pointer.

## Architecture pointers

- **Contract:** `src/shared/types.ts` — IPC channel names + data types; no node/DOM imports (both sides import it). `ConversationMeta` also carries `turnState` / `turnEndedAt` / `lastActivityAt` / `awaitingTool`; `LiveState` is the renderer's resolved working/asking/awaiting/quiet. `IPC.sessionsRename` + `SwitchboardApi.renameConversation(sessionId, title)` are the **only** renderer→main write into a conversation (an empty title clears back to the auto-generated one). `TranscriptBlock` has **no `thinking` variant** (dropped at parse time); `TranscriptMessage` carries `userKind` (`human` / `tool_result` / `interrupted`) so the renderer attributes `user` lines without re-deriving it (see the Transcript-rendering invariant).
- **Main:** `index.ts` (window/lifecycle/smoke, + a dev-only dock icon from `build/icon.png` when unpackaged, + a dev-only window-title suffix from `SWITCHBOARD_DEV_LABEL`), `ipc.ts` (handlers; owns watcher + PtyManager), `sessions/{parser,indexer,watcher,rename}.ts`, `pty/manager.ts`, `menu.ts` (custom app menu — ⌘R→Refresh, no reload roles; see the ⌘R gotcha). `sessions/rename.ts#appendCustomTitle` is the lone JSONL **write** — it appends Claude Code's own `custom-title` line (`fs.appendFile`, O_APPEND, atomic for a small line, safe alongside a live `claude`); the `IPC.sessionsRename` handler appends then **re-indexes + broadcasts `sessionsChanged` immediately** so the new title lands now rather than ~600ms later when the watcher catches the same write. The PtyManager spawns `$SHELL -l -i` then types `claude --resume <id>` / `claude --session-id <id>` (login shell = real PATH; see README "Why a login shell?"). It **gates that boot on the first renderer `resize`**, not just the shell prompt: the PTY spawns at a placeholder 80×30, so booting claude before the real terminal size arrives makes it replay the entire resume transcript at 80×30, and the real size landing mid-replay corrupts claude's cursor math into **real blank rows** in the buffer (which only a later resize/relayout clears). Gating boot on `shellReady && sized` (with a ~2.5s fallback for a terminal that's never shown) makes claude replay at the true size from its first line. PTY busy/idle (output activity, `CONFIG.busyWindowMs` = 800ms) now **only** gates the LRU cap (`CONFIG.maxLivePtys`, **default 8**, user-configurable in Preferences → App (bounds 2–14), persisted by `useMaxLiveSessions` → pushed to the PtyManager over `IPC.ptySetMaxLive`; evicts idle sessions only, never busy work; reaching the cap raises a blocking center-top **capacity modal**, `CapWarningModal`) — it does **not** drive the liveness dot at all (a live TUI repaints constantly — every keystroke echoes as output — so PTY "busy" is ~always true; the transcript turn-state alone drives the dot).
- **Conversation rename + info modal:** two entry points, one surface (`ConversationInfoModal`). The pane title is a button (`PaneHeader`, `.sb-pane-title-btn` — darkened-bg hover, no tooltip) → opens the modal in **view**; a row's right-click **Session details…** (`TallyRail`'s menu, **below** read/unread) also opens it in **view** (the `startInEdit` path remains but no entry point uses it now — both open ready to read, and the title is editable in place). App owns `infoModal: {sessionId, edit}` and renders the modal (resolving the target's meta + pty, mirroring the rail's synth-meta fallback); while it's open App's key handler **short-circuits** (Esc closes, like Preferences / the capacity modal). The conversation title **is** the modal heading, edited **in place**: an always-editable input styled as the heading (`.sb-info-title-input`), commit on **blur**, **Enter** commits *and* closes the modal, **Esc** reverts (a `suppressRef` skips the ensuing blur-commit; `stopPropagation` so App's Esc doesn't also close — first Esc backs out, second closes), and commit writes only on an actual change (never pins an auto-title). A `focusedRef`-guarded `[title]` effect re-seeds the draft from incoming title updates (a commit's re-index, or an external `/rename`) only while the user isn't editing. (There's deliberately **no** "reset to auto title" affordance — it was removed; to drop a custom title, rename it.) Below, the detail rows are grouped into **caps-headed sections** (`Section` → `.sb-info-section` + a `.label-caps` header over a `.sb-info-grid`): **Workspace** (Folder · Branch · Model) · **Activity** (Messages · Size · Duration) · **Tokens** (gated on having any — **Context** first [current window, a live snapshot], then the cumulative per-tier totals **Input · Output · Cache write · Cache read** by Anthropic pricing tier, all visible) · **Session** (live **Status** [the lone accent: a static cobalt dot] / **Started** first, then **Last active** · **Session ID** last). Font split: the **data** (row labels + values) is the **terminal mono face** (`--font-mono` on `.sb-modal-info`) at **one size** (`.sb-info-value` = `--text-xs`, so e.g. Session ID and Started read identically; the `mono` Row prop now only drives truncation, not size), while the **chrome** — the editable title and the section headers (`.sb-info-section-label`) — stays in the **UI face** (Hanken, `--font-ui`/`--font-display`). A deliberate per-dialog split: the readout reads like the terminal it describes, framed by UI-font headers. Values are **selectable** (`.sb-info-value` re-enables `user-select` against the global body `none`) — highlight to copy — and **only Session ID** carries a one-click **`CopyButton`** (shared, extracted from `MessageBlock`) — now the only place to copy the session ID, since the right-click menu's **Copy session ID** item was removed. Timestamps (Last active / Started) show the **absolute** time only; token values are **metric-prefixed** (`format.ts#formatMetric`, e.g. `324K` / `18.4M`), and each token row's **key** (not its value) carries a plain description tooltip via `Row`'s `labelTip` (→ `data-tip` on a span hugging the label, e.g. "Base input tokens") — no raw counts surfaced. The token/size/duration/model values come from `ConversationMeta` fields computed in `extractMeta`'s existing per-line loop (negligible cost): `sizeBytes` · `model` · `firstActivityAt`, plus the token set — `inputBaseTokens` · `cacheWriteTokens` · `cacheReadTokens` · `outputTokens` (cumulative, **deduped by `message.id`** — the same response is written across multiple JSONL lines all repeating identical `usage`, so a naive sum inflates ~3×; `inputTokens` is their input subtotal) and `contextTokens` (the **last main-chain turn's** input+cache = current context-window occupancy, the snapshot Claude Code's status line shows — sidechain turns are counted in the cumulative sums but never set `contextTokens`). The 5m/1h cache-write split exists in the JSONL (`usage.cache_creation.ephemeral_*`) but is folded into `cacheWriteTokens` here. Rename flows `onRename` → `App.renameConversation` → `window.api.renameConversation` → the main append + immediate re-index, so the new title lands in the rail, the pane header, and the still-open modal together.
- **Liveness model:** `sessions/parser.ts#extractTurnState` scans the transcript's main-chain tail (assistant `stop_reason` / a `turn_duration` system line; a streaming fragment has no `stop_reason` yet ⇒ in-progress; a trailing **`AskUserQuestion` / `ExitPlanMode`** tool_use ⇒ `awaiting_input` (+ `awaitingTool` — these tools block for the user's reply, and are always a lone terminal tool_use); sidechain lines ignored) into `ConversationMeta.turnState` + `turnEndedAt` + `lastActivityAt` + `awaitingTool`. **Non-conversational trailing `user` lines are skipped** (`classifyUserLine`): slash-command echo/output (`/model`), `!` bash I/O, background-task notifications, and `isMeta` caveats are NOT a turn — so e.g. running `/model` after a finished turn stays `awaiting`, not falsely `in_progress` (only a real typed prompt or a `tool_result` keeps the turn `in_progress`). An **interrupt sentinel** (`[Request interrupted by user]`, written when you press Esc) ends the turn ⇒ `awaiting`. `resolveLiveState` (extracted to `renderer/lib/liveness.ts`, pure + unit-tested) combines that with a per-session **seen** marker (`useSeen`, localStorage; advanced when a conversation is selected *and* the window is focused — `useWindowFocus`) into the four-state dot: **working** (in-progress) / **asking** (blocked on your reply — AskUserQuestion / ExitPlanMode — unread; clears when looked at, like awaiting) / **awaiting** (finished, unread) / **quiet** (finished+seen, or live-but-not-yet-started — a session with no turn-state yet is quiet, never working). **Manual read/unread:** `useSeen` also holds an `unread` override map (sessionId → markedAt); `resolveLiveState`'s `manualUnread` arg forces `awaiting` (the solid dot) for a finished conversation — or `asking` (the pulse) for one blocked on your reply (`awaiting_input`), so marking a question state unread returns it to pulsing — beating both `lookingNow` and the seen timestamp. Set it with `⇧⌘U` (toggles the selected conversation), **Option+click** — on a live row in the left pane, or anywhere in its live terminal — which **always marks unread, never toggles** (`ConversationRow`'s `onClick` branches on `e.altKey` before its live/non-live dispatch and returns without navigating; the terminal twin is a dedicated capture-phase `mousedown` listener on the `TerminalView` host — `e.altKey && e.button === 0` → `markUnread` — whose `stopPropagation` is load-bearing: it beats `MainPane`'s bubble-phase engage→`markRead` *and* xterm's own mouse handling, and `manualUnread` beating `lookingNow` is what makes the dot go solid even while the conversation is open+focused), or a row's right-click menu (which also offers **Session details…** — on every row, live or not — plus, on a live row only, **Stop session**, which kills the session, and, on a not-live row only, **Resume**, which spawns + focuses it via `App.resumeSession`→`resume`; the read/unread item likewise shows only on live rows); it **auto-expires** once a newer turn lands (`turnEndedAt > markedAt`, derived — no mutation), and is cleared by `markRead` — an explicit toggle, **selecting** the conversation (the seen-effect calls `markRead`, not just `markSeen`), or **engaging** it (a click or keystroke in the pane body, via `MainPane`'s capture-phase listener; scrolling doesn't count). Rows + header show `relTime(lastActivityAt ?? mtime)` (mtime is bumped by resume/metadata writes, so it isn't true last activity). `resolveLiveState` also **demotes a carryover**: an `in_progress` turn whose `lastActivityAt` predates the live process's `PtyState.startedAt` belongs to a prior, now-dead run — you quit/End the app mid-turn (which writes *no* sentinel, unlike Esc), then **resume**, and the new process just replays history and idles at the prompt (without this it would breathe forever). Anything the current process actually does is stamped after `startedAt`, so genuine work — including a long-running tool — is never demoted. Remaining hook-territory gaps (nothing on disk distinguishes them): a turn **blocked on a permission prompt**, and `claude` **crashing while its wrapping shell survives** with no resume (there the turn postdates `startedAt`) — both still read as working.
- **Preload:** `src/preload/index.ts` exposes the typed `window.api` (contextIsolation on, sandbox off for ESM preload). Image input adds `getPathForFile` (Electron `webUtils`, for drag-drop path resolution). Also exposes `window.devLabel` — the dev-only `SWITCHBOARD_DEV_LABEL`, rendered as a mono pill in the title bar (left of the gear) by `TitleBar`.
- **App icon:** `build/icon.svg` is the source; `build/icon.png` + `build/icon.icns` are generated from it (rsvg-convert → sips → iconutil — done manually, no script). `package.json` `build.mac.icon` points at the `.icns`. Regenerate the rasters if you change the SVG.
- **Renderer state (hooks in `src/renderer/lib/`):** `useSessions` (live index), `usePtys` (live processes), `usePins` (localStorage; pinned session IDs as a **manually-ordered array** — `toggle` prepends a new pin / removes, `reorder(from,to)` moves one, and a derived `Set` is exposed for the rail's `.has()` filters; drag-to-reorder commits via `reorder`, pure `reorderArray` unit-tested), `useLiveOrder` (**ephemeral** in-memory — the Live section's manual order; like `usePins` but not persisted, since live PTYs don't outlive the app: every newly-live session, new or resumed, lands on top, drag commits via `reorder` reusing `reorderArray`), `useSeen` (localStorage; per-session "seen" timestamps that split the live dot into unread vs seen, plus an `unread` override map with `markUnread`/`markRead` for manual read/unread), `useWindowFocus` (window focus, which gates when "seen" advances), `useLayout` (localStorage; single-pane shape — pane width + collapsed, and per-section collapse for pinned/live/recent), `useNewConvoDefault` (localStorage; the default-directory-for-new-conversations preference — a folder path + an independent enabled toggle, read by the `+`/⌘N handlers and the Preferences App page), `useMaxLiveSessions` (localStorage; the max-live-sessions cap — default 8, clamped to bounds 2–14; pushed to the main-process PtyManager via `IPC.ptySetMaxLive` and drives the capacity modal's threshold), `useRowReorder` (pointer drag-to-reorder for pinned **and** live rows — one instance per section, partitioned by a CSS `selector` opt; the moving card is a `position:fixed` clone on `<body>` that escapes the rail body's `overflow-y:auto` clip so it floats over the head (its travel clamped to a partial overshoot past the section — see the reorder invariant); the real row is hidden in place as the gap, siblings shift to relocate it, and on drop the clone eases into the settled slot with the FLIP glide suppressed via the reorder nonce), `useTranscript` (live-refreshing transcript with a per-session cache for instant re-show), `useNavHistory` (selection + back/forward history), `useSyncedAnimation` (phase-locks the breathing/ripple liveness animations — and the welcome mark — to a shared `document.timeline` origin so they pulse in unison app-wide; pins each matching CSS animation's `startTime` to 0 in a layout effect, no per-frame cost), `useRailFlip` (FLIP row motion — after an order-changing commit it measures the visible rows' viewport positions and any row that moved between sections glides to its new spot via a compositor-only Web-Animations `translateY`; one measure pass per move, never per render/frame; suppressed during search and section collapse/expand so those stay instant), `useTranscriptSearch` (the find-in-conversation engine — see the Find bullet below), `useAutoHideScrollbar`/`attachAutoHide` (the reveal-on-scroll scrollbar — toggles an `is-scrolling` class, painted by the **shared `.sb-autoscroll` CSS** in `global.css`; one mechanic across the rail body, the New-conversation menu, **and the transcript** — the terminal scrollbar is deliberately scrapped, and there's no fade because Chromium won't animate scrollbar pseudo-elements), `buildGroups` (`messageGroups.ts` — coalesces the transcript into same-source groups; pure + unit-tested), `format` (`relTime` + `clockTime`/`fullDateTime`), `ptyStream` (buffers PTY output to survive the spawn→mount race). The Formatted/Terminal choice is per-conversation state in `App.tsx` (`viewBySession`), clamped to Formatted when no live pty exists.
- **Navigation + focus model — the main pane always owns the keyboard.** There's no list-focus state: the rail never holds the keyboard, and the selected row's emphasis is a single tier (see the two-accent invariant). Selection + back/forward history live in `useNavHistory` (a `useReducer` over `{selectedId, stack, cursor}`): `open` records a history stop on **every** landing (a row click / `⌥⌘↑`/`⌥⌘↓` switch / `⏎` / resume / new); `back`/`forward` (`⌘[` / `⌘]`) walk the stops; `home` (clicking the **Switchboard** wordmark) deselects to the welcome screen — a drift-to-null that leaves history intact, so the next `back` re-centers on the last opened conversation (the reducer's diverge/re-center branch now serves only `home`; the old `preview` action + arrow-drift are gone). History is ephemeral. **Switch conversations with `⌥⌘↑`/`⌥⌘↓`** (Chrome-tab style; `e.code`, per the ⌘⌥ Option-key gotcha): App steps through the visible `orderedIds` (clamped at the ends) and calls `switchTo` — a live target drops into its terminal (`enterLive`: open + Terminal view + focus), a not-live one into its Formatted transcript (`clickConversation`: open + `requestFocus` + a synchronous `paneRef.focus()`, `.sb-pane` is `tabIndex={-1}`). It works **even from inside a live terminal** — Cmd-combos bubble past xterm, the same reason `⌘[`/`⌘]` do. **Focus follows selection:** a `selectedId`-keyed effect `requestFocus`es the landed conversation, so `back`/`forward` (which don't request focus themselves) also land focused; the explicit `requestFocus` in the action handlers still covers re-selecting the *same* conversation (where that effect doesn't fire). **Clicks behave identically and no longer toggle focus** — the old `focusInPaneAtDownRef` mousedown-capture snapshot and the `⌘G`-focus-the-list path are gone. **Focus is decoupled from visibility:** `App` passes a session-targeted `focusReq` (`{sessionId, n}`) down through `MainPane` → `TerminalDeck` → `TerminalView` (`term.focus()` only when the `focusKey` changes — **never** on merely becoming visible) and, in parallel, to `TranscriptView` via `transcriptFocusKey` (`focusReq.sessionId === selectedId ? n : null`), which focuses its scroll container in a layout effect (fires even mid-load), deduped against a ref **owned by `MainPane`** so it survives `TranscriptView` unmounting/remounting (switching across a live terminal unmounts it). **`⏎` from the read-only Formatted transcript** resumes a not-live conversation, or focuses a live one's terminal — gated to the transcript scroll container (`inTranscript`), so pane-header buttons and the live terminal (where xterm consumes Enter) aren't hijacked. Plain `↑`/`↓` are **not** App-level nav — they fall through to the focused surface (scroll the transcript / xterm cursor). `⌘F` still routes on `inMain` (find-in-conversation when the pane is focused, else the rail's cross-conversation search). (`⌘[` / `⌘]` use `e.key`, not `e.code` — the Option-key gotcha is ⌘⌥-only.)
- **Preferences modal + New-menu focus:** `SettingsModal` is a paged **Preferences** modal — a left nav (**App** / **Shortcuts** / **FAQ**) over a scrim — that mirrors the app's surfaces (head + nav `--paper-sunken` = the chrome / unified title-bar + rail; page `--paper-pane` = the main content) and holds a **fixed height** (sized to the taller Shortcuts page; a taller page like FAQ just scrolls its `.sb-settings-page`) so it doesn't resize between pages; dismiss via scrim-click / ✕ / `Esc`. It opens to **App** from the title-bar **gear** or `⌘,` (the macOS Preferences key), and to **Shortcuts** from the footer `?` button in `TallyRail` or `⌘?` (bound Shift-agnostically — `⌘/` *or* `⌘⇧/`). **FAQ** has **no shortcut** — it's reached only by clicking its nav item — and holds a short `FAQ` array (`{q, a}`, prose `.sb-faq`) answering orientation questions: opening Session details, renaming, preview-vs-spawn, the liveness dots, and where the data comes from. While open, `App`'s window key handler short-circuits — `Esc` closes; `⌘,` / `⌘?` toggle (or close) their page — so nothing navigates behind the scrim. The **App** page holds one setting — a **default directory for new conversations** (`useNewConvoDefault`): a folder + an independent **enabled** toggle (so it can be turned off without losing the path; the toggle's on-fill is **ink, not cobalt** — two-accent invariant). When enabled, the rail-head **"+"** and `⌘N` skip the chooser and start straight in that folder (a since-deleted folder rejects in `ipc.ts` via an `existsSync` guard → the renderer falls back to the chooser); the picker reuses `window.api.pickDirectory()`. The **New** menu (`NewConversationMenu`) opens from the rail-head **"+"** (above the search magnifier) or `⌘N` when no default is set, and **always by right-clicking the "+"** (the escape hatch); it focuses its first item on open and roves with `↑`/`↓`/`Home`/`End`, `stopPropagation`-ing arrows + Enter so `App`'s window-level list-nav and Enter-to-resume don't double-fire (its `document` listener sits below `window`), and its recent-dirs list uses the shared auto-hide scrollbar (`useAutoHideScrollbar`).
- **Find-in-conversation + focus-routed `⌘F`:** `⌘F` is focus-sensitive. `App.tsx`'s key handler computes `inMain = paneRef.current?.contains(document.activeElement)` (paneRef is on `.sb-pane`); when focus is in the main pane it opens the in-conversation find bar, otherwise it falls through to the rail's cross-conversation fuzzy search (`fuzzy.ts`, unchanged) — whose field is boxed like the rename title input (`.sb-rail-search-box`/`-input`; the box persists the whole time search is open, not just on focus), with the rail magnifier shown only while closed and, once open, replaced by an inset ✕ (`.sb-rail-search-clear`) at the field's right edge. **That same `inMain` test also gates the Arrow/Enter list-nav** — so once you click into the transcript or terminal, the main pane owns the keyboard (the transcript is `tabIndex={-1}` so a click focuses it; arrows then scroll it instead of moving the list selection). The find bar (`TranscriptSearch`) renders **inline in the pane header**; the magnifier in the header action cluster (left of the pin) **opens** it and is **hidden while it's open** — the find bar's own ✕ closes it, landing roughly where the magnifier was so the two never both show, and the bar itself carries no leading magnifier icon. Pressing `⌘F` while the bar is already open (e.g. after clicking into the transcript) **re-focuses** its input — `App` bumps a `findFocusReq` counter threaded App→MainPane→PaneHeader→TranscriptSearch, and `TranscriptSearch` focuses + selects on each change (the input otherwise only auto-focuses on mount, so a second `⌘F` would be a no-op). Find state lives in `MainPane` (`findQuery` / `activeIndex` / `matchCount`; the query is `useDeferredValue`-debounced). The engine is **`useTranscriptSearch`** (in `TranscriptView`): a `TreeWalker` over `.transcript-content` builds a `Range` for each case-insensitive substring (the pure matcher `findMatches` is unit-tested), painted via the **CSS Custom Highlight API** (`::highlight(sb-find)` / `sb-find-current`, `transcript.css`) — **no DOM mutation**, so it doesn't fight react-markdown or defeat `MessageBlock`'s memo (the snappiness-first choice). Coverage now **includes collapsed sections**: matches inside closed tool `<details>` and the clamped tail of long results are found + counted, and navigating to one **reveals** it first — `revealContainer` opens the native `<details>` directly and asks a clamped result to expand via an `sb-reveal` event (then defers the scroll one frame for the relayout). Only transcript chrome (footer, loading skeleton) is excluded; matches are still found within a single text node. Opening find over a live **Terminal** switches to Formatted on the first keystroke and restores the prior view on close. The rail's live-session count comes from an **all-sessions** tally in App (`liveTally` over `ptys.active`), never the search-filtered rail set — so filtering the list never makes the count drop.
- **Tooltips (`data-tip` + `TooltipLayer`):** app-wide hover labels are a single fixed-positioned element (`TooltipLayer`, mounted at the app root) driven by `data-tip` attributes via document-level event delegation — it replaced native `title` everywhere (native `title` lags ~1s and resets on the slightest pointer move, so it rarely showed). Fixed positioning escapes scroll/overflow containers (the rail list, the modal's `overflow:hidden`), flips above when near the bottom edge, and clamps to the viewport at the sides. Look is `.sb-tip` in `global.css` (ink-on-paper — no accent). Add a tooltip by putting `data-tip="…"` on any element; icon-only buttons keep an `aria-label` for the accessible name. (The only remaining `title=` in the renderer are two React *props* passing the conversation-title string, not HTML attributes.)
- **Theming (`useTheme` → `theme.ts` + `themeDom.ts`):** the app theme is `useTheme` (owned once in `App`; `localStorage` `switchboard.theme` = `system`|`light`|`dark`, default **system**). `theme.ts` is the **pure** core — `ThemeMode` / `ResolvedTheme` / `resolveTheme` — so the test suite imports it under the **node** tsconfig (which has no DOM lib); the browser-side helpers (`readThemeMode`, `systemPrefersDark`, `applyTheme`) live in `themeDom.ts` (renderer-only). Keep that split — putting a `window`/`document` reference in `theme.ts` breaks `typecheck:node`, since `test/` pulls test-imported renderer libs into the node project. `applyTheme` writes `data-theme` on `<html>` and reads the **computed** `--paper` to sync the window `backgroundColor` (drift-proof against token retunes). `main.tsx` calls `applyTheme(resolveTheme(readThemeMode(), systemPrefersDark()))` **before `root.render`** — the window is hidden until first paint, so this kills the FOUC without a CSP-blocked inline `<head>` script. `system` tracks `matchMedia('(prefers-color-scheme: dark)')` live. The resolved theme threads `App → MainPane → TerminalDeck → TerminalView` (same path as `focusReq`); `TerminalView` holds two xterm theme objects (`LIGHT_THEME` / `DARK_THEME`, a neutral 16-color ANSI palette + cobalt selection) and swaps `term.options.theme` live on change, so a running `claude` recolors in place (the construction effect captures the initial theme; a `[theme]` effect owns changes). Controls: a **sun/moon** title-bar toggle (`TitleBar`, flips light↔dark as an explicit choice) and the Preferences **App → Appearance** System/Light/Dark segmented control (reuses `.sb-seg`). Window-bg sync rides `IPC.windowSetBackgroundColor` (`shared/types` + preload `setBackgroundColor` + an `ipcMain.on` calling `BrowserWindow.fromWebContents(e.sender)?.setBackgroundColor`); `main/index.ts` seeds the initial `backgroundColor` from `nativeTheme.shouldUseDarkColors` for the hidden pre-paint frame.

## Conventions

- TypeScript strict; 2-space indent; no semicolons (match `src/shared/types.ts`). Prefer named exports. No formatter is configured.
- Main/preload imports are relative; the renderer uses the `@shared` / `@renderer` aliases.
- Repo-wide test/build is cheap here; still prefer the filtered loop above.
