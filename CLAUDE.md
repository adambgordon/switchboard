# Switchboard ‚Äî project guide

A macOS Electron app ‚Äî light or dark theme ‚Äî that browses Claude Code conversations: read-only transcript preview, resume/new into a real terminal (node-pty + xterm.js), a live "Formatted" view, and pinning. The layout is two columns ‚Äî a single unified left pane (Pinned / Live / Recent conversations) and a main pane. **All conversation data is derived from the JSONL files Claude Code writes under `~/.claude/projects/`** ‚Äî Switchboard never owns it.

Stack: Electron 42 (pinned as `latest`, so it tracks upstream) ¬∑ electron-vite ¬∑ React 18 + Vite ¬∑ TypeScript (strict) ¬∑ plain CSS ¬∑ node-pty ¬∑ chokidar ¬∑ @xterm/xterm ¬∑ react-markdown ¬∑ fuse.js. No Tailwind, no state library (plain hooks + localStorage).

## Set up Switchboard (first run)

When the user asks to **"Set up Switchboard"** (typically from a fresh clone ‚Äî this is the phrase the public README's "Set up" section points them at), run the one-shot setup and then open the built app:

```bash
npm run setup
open dist/mac-arm64/Switchboard.app
```

`npm run setup` (= `scripts/setup.mjs`) installs dependencies, rebuilds `node-pty` for Electron's ABI, and packages `dist/mac-arm64/Switchboard.app` ‚Äî but it does **not** launch the app, so always follow it with `open`. If it fails on a missing prerequisite (Xcode Command Line Tools, or Node 26 ‚Äî see `.nvmrc`), walk the user through installing it, then re-run.

## Build, validate, run

The building blocks (run individually as needed):
1. `npm run typecheck` ‚Äî `tsc` over BOTH projects (main = node, renderer = web). Must be clean.
2. `npm run build` ‚Äî `electron-vite build` ‚Üí `out/`. Must succeed. **Does NOT update the `.app`** ‚Äî `out/` is what `npm run dev` and the smoke check consume, not the packaged bundle.
3. **Boot smoke:** `SWITCHBOARD_SMOKE=1 node_modules/.bin/electron .` ‚Äî prints `SMOKE PASS | pty:true | window:loaded`. The only headless check that node-pty loads/spawns under Electron's ABI and the renderer loads. `window:none` is a sampling race (re-run); only `did-fail-load` / `preload-error` lines indicate a real failure.
4. `npm test` ‚Äî vitest (parser / indexer / customTitle). The renderer is not unit-tested.
5. `npm run package` ‚Äî `electron-vite build` + `electron-builder --dir`, rebuilding the installed `dist/mac-arm64/Switchboard.app`. `npm run build` alone does NOT do this. Quit (‚åòQ) and reopen the app to load it.

For iterating, `npm run dev` (hot reload) is the inner loop. The renderer (visual feel, live terminal, drag/scroll) can only be truly verified by a human running `npm run dev` or the freshly-packaged `.app` ‚Äî typecheck/build/smoke don't catch React-runtime or visual issues. Say so rather than claiming the UI works.

Set `SWITCHBOARD_DEV_LABEL=<name>` before `npm run dev` to tag the window title **and** the title bar (a mono pill left of the gear) ‚Äî so several dev instances can run side by side and stay tellable apart (there's no single-instance lock; Vite auto-increments the port). Unset in normal and packaged runs, where the title stays a plain "Switchboard".

## Gotchas (these have bitten before)

- **node-pty rebuild:** use `npm run rebuild` (= `node scripts/rebuild-native.mjs`). The `@electron/rebuild` CLI crashes under Node 26 (yargs/ESM), so we call its programmatic API directly.
- **Packaging:** if you have no code-signing identity, package with `CSC_IDENTITY_AUTO_DISCOVERY=false`. macOS may Gatekeeper-prompt on first open of a fresh build.
- **localStorage is per-origin:** dev (`http://localhost`) and the packaged `.app` (`file://`) don't share pins/layout. Expected, not a bug.
- **macOS Option key:** for `‚åò‚å•` shortcuts use `e.code` (e.g. `'KeyB'`), not `e.key` (Option+B yields `‚à´`).
- **Terminal image input rides on bracketed paste.** claude attaches an image on a *paste* ‚Äî it reads the system clipboard, or parses an image path inside the pasted text ‚Äî never on plain typed text. So `TerminalView` delivers via the bracketed-paste sequence (`ESC[200~ ‚Ä¶ ESC[201~`): **Ctrl+V** sends an *empty* one (claude then reads the clipboard image ‚Üí `[Image #N]`); **drag-drop** wraps the dropped, shell-escaped path(s) in one. The Cmd+V/Ctrl+V roles are **inverted vs. a native terminal**: under xterm, **Cmd+V** is the real (browser) `paste` event while **Ctrl+V** would only send `\x16`. So we own **Ctrl+V** via `attachCustomKeyEventHandler`, and intercept **Cmd+V** in *capture phase* to keep it text-only (an image-only clipboard is suppressed, so claude can't grab it). macOS-only (Cmd+V fires a DOM `paste` event; Ctrl+V doesn't).
- **Terminal renders via the WebGL addon (canvas is the fallback), never the DOM renderer.** `TerminalView` loads `@xterm/addon-webgl` *after* `term.open()` (required ordering); on GPU-context loss (`onContextLoss`) ‚Äî or if WebGL can't initialize ‚Äî it swaps to `@xterm/addon-canvas`. WebGL is the faster, higher-fidelity renderer (it draws every glyph each frame *without* clipping it to the cell, which canvas does ‚Äî that clipping was the source of the box-glyph/whitespace artifacts) and is where xterm.js + VS Code are consolidating. Both honor `customGlyphs` (default on), so box-drawing/block glyphs paint as continuous vector lines. The fallback is **canvas, not the default DOM renderer**: DOM pulls box-glyphs from the font, which breaks `claude`'s table/box borders at our non-integer `fontSize` (11.5) + `lineHeight` (1.3). Don't switch the *primary* off WebGL or the *fallback* to DOM without eyeballing how tables render. (Text stays IBM Plex Mono either way.)
- **Terminal `lineHeight` is kept near-integer for the canvas fallback.** `11.5 √ó 1.3 ‚âà 15px` makes the cell height ~integer. The WebGL primary doesn't need this, but the **canvas fallback** strands blank rows during rapid live redraws at a fractional cell (claude's tool-call output) ‚Äî intermittent vertical whitespace gaps that only a repaint (scroll, or ‚åò +/- zoom) clears, never selection (selection is a separate render layer). It was `1.32` (15.18px); WebGL fixed those gaps at the root (no per-cell clipping / repaint desync), but keep `1.3` so the fallback path stays clean. Don't raise `lineHeight` to a value that makes the cell fractional again.
- **Terminal Unicode width is pinned to v11.** `TerminalView` loads `@xterm/addon-unicode11` and sets `term.unicode.activeVersion = '11'` right after `term.open()` (before the first PTY write). xterm's built-in default is the Unicode **V6** width table, which measures emoji-presentation chars (‚úÖ `U+2705`, ‚ùå `U+274C`, ‚Ä¶) as **1 cell** ‚Äî but `claude` lays out tables / box-art assuming the modern **2-cell** emoji width (as every real terminal does), so the 1-cell mismatch shifted every cell after an emoji and knocked the closing `‚îÇ` border out of alignment (and the half-cell glyph overdraw flickered under partial canvas repaints on scroll/select). V11 widens those codepoints to 2, matching claude. Requires `allowProposedApi: true` (the `term.unicode` interface is proposed API). Don't remove the addon or fall back to V6 ‚Äî and don't confuse this with grapheme clustering, a separate, costlier feature we deliberately don't enable (V11 is just a corrected width table; the per-char hot path is unchanged from V6).
- **Terminal cursor is hidden ‚Äî Switchboard defers to Claude's own caret.** `TerminalView` sets `theme.cursor` transparent (`rgba(0,0,0,0)`) with `cursorBlink: false`. `claude` (like most full-screen TUIs) draws and styles its OWN caret ‚Äî a reverse-video block ‚Äî so xterm's hardware cursor would just paint a redundant *second* cursor on the same cell (it showed as a cobalt bar beside Claude's block). Transparent leaves Claude's block as the only caret. `cursorStyle: 'bar'` is kept solely so the invisible cursor never recolors the glyph under it (a `block` cursor would), and this subsumes the old `cursorInactiveStyle: 'none'` too (a transparent cursor is invisible focused or not, so arrow-preview shows no stray box). Trade-off: no visible cursor at a bare shell prompt ‚Äî fine, since this terminal exists to host a live `claude`. Cursor *style* is therefore not a Preference (deliberately not among the App page's settings) ‚Äî it's Claude's to draw.
- **A hidden terminal's renderer is paused, so it's force-repainted on show.** xterm's `RenderService` installs an `IntersectionObserver` on the screen element and sets `_isPaused` while it isn't intersecting the viewport ‚Äî which `display:none` triggers ‚Äî after which `refreshRows` **drops** every paint request. `TerminalDeck` hides inactive live sessions with `display:none`, so a session that animates while hidden (claude's mid-thought spinner) freezes on its last painted frame when you navigate to it, and stays frozen until a redraw is forced (a manual scroll was the only cue). `TerminalView`'s becomes-visible effect fixes it: it **clears that paused flag directly** (`term._core._renderService._isPaused = false`, reached via private API and guarded with optional chaining) before the fit + `term.refresh(0, rows-1)`. The earlier fix ‚Äî refreshing on two `requestAnimationFrame`s and hoping the observer had un-paused ‚Äî was unreliable, because Chromium does **not** consistently re-fire the `IntersectionObserver` on a pure `display:none ‚Üí block` toggle (it recomputes intersection on scroll/layout), so the refresh got dropped while still paused (symptom: a frozen spinner that only a manual scroll woke). Clearing the flag ourselves guarantees the repaint; the observer later sets it correctly (visible ‚áí false), a no-op. One-time per navigation (no per-frame cost). Don't drop it ‚Äî and don't "fix" it by taking the deck off `display:none`, which would keep every hidden live terminal repainting (a CPU regression against the LRU/pause bounds).
- **Never fit/resize a hidden (`0√ó0`) terminal ‚Äî it pushes a bogus tiny size that strands blank rows.** When a deck item goes `display:none` (you switch to another conversation), its host measures `0√ó0` *and* its `ResizeObserver` fires ‚Äî but `@xterm/addon-fit` does **not** bail on a zero-size host: it reads the host's `height`/`width:100%` *computed* style (`parseInt("100%")` ‚Üí 100px) and proposes a ~`13√ó5` grid. Pushing that to the PTY makes `claude` reflow its whole TUI into 13√ó5 and then back to full size when you return, and the reflow strands **real blank rows** in the buffer (the "whitespace after tool calls" gap ‚Äî same symptom as the 80√ó30 boot bug, but a *different* cause: a live resize, not boot replay). `TerminalView` funnels every resize through one `fitAndResize` that **skips entirely when `host.clientWidth`/`clientHeight === 0`** and **suppresses no-op resizes** (sends only when `cols`/`rows` actually changed), so switching conversations/views sends *zero* resizes when the size is unchanged. Don't reintroduce an unconditional `window.api.resize(...)` after `fit.fit()`, and don't drop the `0√ó0` guard ‚Äî the `ResizeObserver` firing on hide is the trigger. (The guard also subsumes the boot case: a hidden mount no longer sends xterm's 80√ó24 default, so it can't satisfy the boot-size gate at the wrong size.)
- **The initial terminal fit runs before the PTY backlog, on purpose.** `TerminalView` calls `fitAndResize()` once right after `term.open()` + addons and *before* `attachPty`, so the expensive WebGL resize hits an empty terminal instead of a screenful mid-replay ‚Äî it removed a ~170ms main-thread stall on resume (trace-verified). The rAF + `ResizeObserver` after it are the fallback (host not yet measurable) and the later-resize path; don't move the first fit back behind the flood. The residual *choppy scroll for the first seconds after resume* is GPU/compositor frame-pacing ‚Äî not main-thread, not cheaply fixable ‚Äî so don't re-chase it as a main-thread problem.
- **Fonts are warmed before first paint.** `main.tsx` gates `root.render` on `document.fonts.load(...)` for every bundled weight (Hanken 400/500/600/700/800, IBM Plex Mono 400/500/600), raced against a 1500 ms ceiling. `@fontsource` ships each weight as a separate `font-display: swap` file fetched lazily on first use, so a bold run could paint in a fallback face and then **not** repaint when the real face loaded ‚Äî Chromium leaves the already-painted glyphs stale until a reflow (selecting or zooming the text forced it). Decoding up front closes that window: text paints once, in the right face (~20 ms on local files, behind the `show:false` window). Don't drop the gate or the intermittent bold-font flash returns ‚Äî and note `document.fonts.ready` *alone* is a no-op here (nothing is pending before render, so the explicit `.load()` calls are what kick off the decode).
- **Don't clamp `white-space: pre-wrap` text with `-webkit-line-clamp`.** The Formatted view caps a long tool result at **12 rendered lines** via a content-box `max-height` (= 12 line-heights, `1.55em` each) on a **padding-less inner clip** element with `overflow:hidden`; the gap below the cut is the panel's own bottom padding. `-webkit-line-clamp` was tried (twice) and fails on `pre-wrap`: it cuts **mid-line** AND makes `scrollHeight === clientHeight`, which broke the overflow check that gates "Show more". That check is now a `ResizeObserver` on the text (a one-shot read raced layout + font-load and dropped the button).

## Design invariants (don't break these)

- **Two accent colors, each with one job.** Cobalt (`--live`, `#1F5AE6`) = a session is *alive*; the only *action* that may carry it is the one that brings a conversation to life ‚Äî the right-click **Resume** (cobalt-on-hover, the live counterpart to Stop's red). A warm red (`--danger`, `#d8392c`) = *negative / destructive* (End/Stop controls, error states). Everything else ‚Äî pins, selection, metadata ‚Äî stays ink/graphite.
- **Row controls live in the right gutter.** In the left pane's rows, the liveness indicator and the pin toggle share one right-hand slot: at rest a live row shows the cobalt liveness dot (everything else is blank), whose form encodes the conversation's turn-state ‚Äî **breathing** = a turn is in progress, **a cobalt ripple** (rings pulsing outward from a briefly-swelling dot) = Claude is blocked on your reply (an AskUserQuestion / ExitPlanMode, unread), **solid** = the turn finished and you haven't viewed it since (unread), **hollow ring** = finished and already seen (up to date); the solid/unread state can also be set **by hand** (`‚áß‚åòU`, **Option+click** on a live row, or a row's right-click menu ‚Äî see the Liveness model below); hovering a **non-live** row reveals the pin/unpin toggle (filled = pinned, outline = not); on a **live** row the pin appears only when you hover the **gutter** itself (the dot's slot), which fades the dot out behind it ‚Äî so skimming a live row's body never disturbs its liveness dot. The pin is **never shown at rest, even when pinned** ‚Äî the **Pinned** section already says it's pinned. Rows no longer carry an inline Stop; a live session is ended by the **Stop** button in the pane header, or a live row's right-click ‚Üí **Stop session**. (The pane header keeps its own pin on the **left** of the action cluster; there, the **Terminal** view-toggle carries a **static solid cobalt dot** ‚Äî it marks that the session is live but no longer encodes turn-state; the breathing/ripple liveness animation lives only on the left-pane rows.) **Pinned and Live** rows can each be **click-dragged** vertically to reorder within their own section (`useRowReorder`, one instance per section partitioned by CSS selector ‚Äî `.sb-row.pinned` vs `.sb-row.live:not(.pinned)`, so a pinned-live row is owned by the Pinned instance; the moving card is a `position:fixed` clone on `<body>` so it floats over the head instead of being clipped by the rail body's overflow; its vertical travel is **clamped to a partial overshoot** past its section (‚âà half a row ‚Äî the card's center can't leave the section), not free; siblings slide live to open the drop gap, fixed section height; on drop the clone eases into its slot *while* dropping out of the lifted look at the same time, so release reads as one motion, not a slide-then-snap-flat); reorder is always on.
- **The pane orders by stable keys, never live activity.** Rows hold position by pin order (**Pinned**) and a **manual order** (**Live** ‚Äî every newly-live session, new or resumed, lands on top, and you drag to reorder; held in `useLiveOrder`, ephemeral) ‚Äî never `lastActivity` ‚Äî so a session emitting output never makes a live row jump (regression fixed in `ce407fa`; the manual order now makes this structural ‚Äî Live no longer auto-sorts by `startedAt` at all, a row holds its slot until you move it). The **Recent** section is the deliberate exception: a recency history of *not-live* conversations, sorted by `mtime`. Pinning a row jumps it to the top of **Pinned** ‚Äî but that's a user action, not activity. **The whole pane renders through `ConversationRow`** ‚Äî the unified left pane *is* the left column now (no separate sidebar): card chrome (`.sb-row.card`) marks **live rows only** (a white card = a live session, so a non-live pinned row renders flat like Recent), and Pinned/Live add the live click-to-jump; the pin-on-hover gutter toggle is a shared `.sb-row` primitive, so it (and the liveness dot) work on Recent's flat rows too. The three sections are individually collapsible, and **Recent** is capped (30) with a *Show more* reveal at the bottom. So `ConversationRow` changes flow into every section. When a row *does* change section ‚Äî pinned, resumed, or unpinned ‚Äî it **glides** to its new position via a FLIP animation (`useRailFlip`) instead of teleporting; the old `sb-patch-in` entrance keyframe was retired (it re-fired when a row gained `.card` in place, collapsing then re-expanding a resumed pinned row ‚Äî the "delete + re-add" flicker), so a row going live just eases its card *background* in. `border-color` and `box-shadow` are deliberately **not** transitioned on `.card` rows, so the cobalt selection ring (`.card.live.selected`) is **instant** on select *and* deselect ‚Äî matching the flat rows, which snap the same way; transitioning them made the blue linger ~120ms (an expo-out tail) when you switched to another conversation. (Card hover snaps too, as a consequence ‚Äî consistent with the rail's flat rows.)
- **Single-family type:** Hanken Grotesk for UI + display (heavy weights carry the display moments); IBM Plex Mono for the terminal + mono metadata. No serif. (One narrow exception: keyboard-shortcut glyphs ‚Äî `‚åò ‚áß ‚å• ‚èé ‚Üë ‚Üì` ‚Äî render in `--font-keys` = the macOS system font, which carries them at consistent size/baseline; Hanken & Plex lack several and fall back per-glyph, which looked small/off-baseline.)
- **Preview ‚âÝ spawn:** selecting a conversation only previews it (read-only, from the JSONL). A live `claude` process is created ONLY by explicit **Resume** or **New**.
- **Formatted view is read-only** (rendered from the JSONL); typing happens in the **Terminal** view. It is also the surface for **find-in-conversation** (`‚åòF` when the main pane is focused) ‚Äî and find now **reveals** collapsed sections (opens a closed tool `<details>`, expands a clamped result) when you navigate to a hidden match; only transcript chrome is excluded.
- **Transcript rendering is one `MessageBlock` per *group*, not per message.** `lib/messageGroups.ts#buildGroups` coalesces consecutive same-source lines under a single header ‚Äî Claude ‚Üí Claude ‚Üí Result ‚Üí Result renders as one **Claude** then one **Result** (memoized on the transcript so search re-renders keep `MessageBlock`'s memo). **Attribution is by content, not raw role:** the parser tags each `user` line `userKind` ‚Äî `human` (typed prompt ‚Üí "You"), `tool_result` (tool output ‚Üí a **Result**/**Error** header, never "You"), or `interrupted` (the Esc sentinel ‚Üí a centered muted note); non-conversational `user` lines (slash-command echo, `isMeta`, task-notifications) **and thinking blocks** are dropped at parse time. A single hairline rule separates a **You** (human) turn from the surrounding Claude / Result / Error turns ‚Äî drawn ONLY at that boundary, never between same-side turns (`.message.has-divider`, computed in `TranscriptView` as `dividerBefore`). The **You** body renders as plain `white-space:pre-wrap` prose (same ink/size as Claude's text, no box, **not** markdown-parsed ‚Äî so typed line breaks survive). Tool calls/results are **boxless** ‚Äî the `‚öô Name` head is itself the click-to-expand disclosure (gear only, no chevron, hover-lightens, `user-select:none`) over the sunken JSON/result block ‚Äî **except an error result, which keeps its red box**. Each header carries a quiet right-aligned timestamp (`format.ts#clockTime`; full date/time on hover via `data-tip`). Code/mono surfaces sit at the **terminal size** (`--text-xs`).
- Design tokens live in `src/renderer/styles/tokens.css` ‚Äî reuse the CSS vars; don't hardcode colors.
- **Two themes, one source of truth ‚Äî a hardcoded color is a dark-mode bug, not just a style nit.** `tokens.css` defines a **neutral grayscale** light palette in `:root` and a **neutral near-black** dark palette in `:root[data-theme='dark']` (only color + shadow tokens are overridden; type/spacing/radii/motion/layout are shared). Surface tokens: `--paper-sunken` = **chrome** (the rail AND the title bar ‚Äî `.sb-titlebar` uses `--paper-sunken`, not `--paper`, so they read as one continuous chrome), `--paper-pane` = **main content** (pane / Formatted / terminal), `--paper-raised` = **raised tiles** (live cards, modals, menus), `--paper` = the field/shell behind them. **Dark deliberately INVERTS the elevation** (see the dark-scope comment): the chrome (`--paper-sunken`) is *lighter* than the content canvas (`--paper-pane`), and tiles (`--paper-raised`) lighter still ‚Äî the reverse of light mode and of the usual dark convention. Every surface paints from a token, so dark mode is "free" for any component that obeys this. The **two-accent invariant holds in both themes**: cobalt = alive, red = destructive (both brightened for dark, soft/wash tints dark-tinted); pins/selection/metadata stay ink. One-off shadows use the `--shadow-color` channel (`rgba(var(--shadow-color), Œ±)`) and overlays use `--scrim`. The Preferences modal mirrors these surfaces (chrome head+nav, content page ‚Äî see the Preferences pointer). The active theme is a JS-resolved `data-theme` on `<html>` (System/Light/Dark), **never** a bare `@media (prefers-color-scheme)` ‚Äî so explicit Light/Dark beats the OS. See the **Theming** architecture pointer.

## Architecture pointers

- **Contract:** `src/shared/types.ts` ‚Äî IPC channel names + data types; no node/DOM imports (both sides import it). `ConversationMeta` also carries `turnState` / `turnEndedAt` / `lastActivityAt` / `awaitingTool`; `LiveState` is the renderer's resolved working/asking/awaiting/quiet. `TranscriptBlock` has **no `thinking` variant** (dropped at parse time); `TranscriptMessage` carries `userKind` (`human` / `tool_result` / `interrupted`) so the renderer attributes `user` lines without re-deriving it (see the Transcript-rendering invariant).
- **Main:** `index.ts` (window/lifecycle/smoke, + a dev-only dock icon from `build/icon.png` when unpackaged, + a dev-only window-title suffix from `SWITCHBOARD_DEV_LABEL`), `ipc.ts` (handlers; owns watcher + PtyManager), `sessions/{parser,indexer,watcher}.ts`, `pty/manager.ts`. The PtyManager spawns `$SHELL -l -i` then types `claude --resume <id>` / `claude --session-id <id>` (login shell = real PATH; see README "Why a login shell?"). It **gates that boot on the first renderer `resize`**, not just the shell prompt: the PTY spawns at a placeholder 80√ó30, so booting claude before the real terminal size arrives makes it replay the entire resume transcript at 80√ó30, and the real size landing mid-replay corrupts claude's cursor math into **real blank rows** in the buffer (which only a later resize/relayout clears). Gating boot on `shellReady && sized` (with a ~2.5s fallback for a terminal that's never shown) makes claude replay at the true size from its first line. PTY busy/idle (output activity, `CONFIG.busyWindowMs` = 800ms) now **only** gates the LRU cap (`CONFIG.maxLivePtys`, **default 8**, user-configurable in Preferences ‚Üí App (bounds 2‚Äì16), persisted by `useMaxLiveSessions` ‚Üí pushed to the PtyManager over `IPC.ptySetMaxLive`; evicts idle sessions only, never busy work; reaching the cap raises a blocking center-top **capacity modal**, `CapWarningModal`) ‚Äî it does **not** drive the liveness dot at all (a live TUI repaints constantly ‚Äî every keystroke echoes as output ‚Äî so PTY "busy" is ~always true; the transcript turn-state alone drives the dot).
- **Liveness model:** `sessions/parser.ts#extractTurnState` scans the transcript's main-chain tail (assistant `stop_reason` / a `turn_duration` system line; a streaming fragment has no `stop_reason` yet ‚áí in-progress; a trailing **`AskUserQuestion` / `ExitPlanMode`** tool_use ‚áí `awaiting_input` (+ `awaitingTool` ‚Äî these tools block for the user's reply, and are always a lone terminal tool_use); sidechain lines ignored) into `ConversationMeta.turnState` + `turnEndedAt` + `lastActivityAt` + `awaitingTool`. **Non-conversational trailing `user` lines are skipped** (`classifyUserLine`): slash-command echo/output (`/model`), `!` bash I/O, background-task notifications, and `isMeta` caveats are NOT a turn ‚Äî so e.g. running `/model` after a finished turn stays `awaiting`, not falsely `in_progress` (only a real typed prompt or a `tool_result` keeps the turn `in_progress`). An **interrupt sentinel** (`[Request interrupted by user]`, written when you press Esc) ends the turn ‚áí `awaiting`. `resolveLiveState` (extracted to `renderer/lib/liveness.ts`, pure + unit-tested) combines that with a per-session **seen** marker (`useSeen`, localStorage; advanced when a conversation is selected *and* the window is focused ‚Äî `useWindowFocus`) into the four-state dot: **working** (in-progress) / **asking** (blocked on your reply ‚Äî AskUserQuestion / ExitPlanMode ‚Äî unread; clears when looked at, like awaiting) / **awaiting** (finished, unread) / **quiet** (finished+seen, or live-but-not-yet-started ‚Äî a session with no turn-state yet is quiet, never working). **Manual read/unread:** `useSeen` also holds an `unread` override map (sessionId ‚Üí markedAt); `resolveLiveState`'s `manualUnread` arg forces `awaiting` (the solid dot) for a finished conversation ‚Äî or `asking` (the pulse) for one blocked on your reply (`awaiting_input`), so marking a question state unread returns it to pulsing ‚Äî beating both `lookingNow` and the seen timestamp. Set it with `‚áß‚åòU` (toggles the selected conversation), **Option+click** ‚Äî on a live row in the left pane, or anywhere in its live terminal ‚Äî which **always marks unread, never toggles** (`ConversationRow`'s `onClick` branches on `e.altKey` before its live/non-live dispatch and returns without navigating; the terminal twin is a dedicated capture-phase `mousedown` listener on the `TerminalView` host ‚Äî `e.altKey && e.button === 0` ‚Üí `markUnread` ‚Äî whose `stopPropagation` is load-bearing: it beats `MainPane`'s bubble-phase engage‚Üí`markRead` *and* xterm's own mouse handling, and `manualUnread` beating `lookingNow` is what makes the dot go solid even while the conversation is open+focused), or a row's right-click menu (which also offers **Copy session ID** ‚Äî on every row, live or not ‚Äî plus, on a live row only, **Stop session**, which kills the session, and, on a not-live row only, **Resume**, which spawns + focuses it via `App.resumeSession`‚Üí`resume`; the read/unread item likewise shows only on live rows); it **auto-expires** once a newer turn lands (`turnEndedAt > markedAt`, derived ‚Äî no mutation), and is cleared by `markRead` ‚Äî an explicit toggle, **selecting** the conversation (the seen-effect calls `markRead`, not just `markSeen`), or **engaging** it (a click or keystroke in the pane body, via `MainPane`'s capture-phase listener; scrolling doesn't count). Rows + header show `relTime(lastActivityAt ?? mtime)` (mtime is bumped by resume/metadata writes, so it isn't true last activity). `resolveLiveState` also **demotes a carryover**: an `in_progress` turn whose `lastActivityAt` predates the live process's `PtyState.startedAt` belongs to a prior, now-dead run ‚Äî you quit/End the app mid-turn (which writes *no* sentinel, unlike Esc), then **resume**, and the new process just replays history and idles at the prompt (without this it would breathe forever). Anything the current process actually does is stamped after `startedAt`, so genuine work ‚Äî including a long-running tool ‚Äî is never demoted. Remaining hook-territory gaps (nothing on disk distinguishes them): a turn **blocked on a permission prompt**, and `claude` **crashing while its wrapping shell survives** with no resume (there the turn postdates `startedAt`) ‚Äî both still read as working.

- **Renderer state (hooks in `src/renderer/lib/`):** `useSessions` (live index), `usePtys` (live processes), `usePins` (localStorage; pinned session IDs as a **manually-ordered array** ‚Äî `toggle` prepends a new pin / removes, `reorder(from,to)` moves one, and a derived `Set` is exposed for the rail's `.has()` filters; - **Navigation + focus model:** selection and back/forward history live in `useNavHistory` (a `useReducer` over `{selectedId, stack, cursor}`). `open` records a history stop (a live-row click / `‚èé` / `‚Üí` on a live row / resume / new); `preview` moves the selection only (`‚Üë`/`‚Üì` ‚Äî **no** stop); `back`/`forward` (`‚åò[` / `‚åò]`) walk the stops, and after an arrow-drift off the current stop the first `back` re-centers on it; `home` (clicking the **Switchboard** wordmark in the title bar) deselects to the welcome screen ‚Äî a drift-to-null that leaves history intact, so the next `back` re-centers on the last opened conversation. History is ephemeral (not persisted). **Terminal focus is decoupled from visibility:** `App` passes a session-targeted `focusReq` (`{sessionId, n}`, `n` bumped only on the **focus-grabbing** actions ‚Äî a **live-row click** / `‚èé` / `‚Üí` on a live row / resume / new / go-live, **not** `‚Üë`/`‚Üì` preview or back/forward) down through `MainPane` ‚Üí `TerminalDeck`, which hands the matching `TerminalView` a numeric `focusKey`; `TerminalView` calls `term.focus()` only when that key changes ‚Äî **never** on merely becoming visible. So **clicking a live row you're not already in** opens it *and* focuses its terminal ‚Äî the click runs `clickLive`, which for a *different* row takes the same `enterLive(id?)` path as `‚èé` / `‚Üí` (open + Terminal view + focus) so you can type immediately; **clicking the row you're already in ALWAYS releases** the keyboard to the list ‚Äî `clickLive` is just `if (id !== selectedId) enterLive(id)`, so a same-row click is a no-op and the click's own blur hands focus from the terminal back to the rail (arrow-nav resumes). It **never re-grabs** (the old click-to-toggle, and its `focusInPaneAtDownRef` mousedown snapshot, are gone). `‚èé` / `‚Üí` still call `enterLive` directly (always focus). **Arrow `‚Üë`/`‚Üì` preview** shows a live row's terminal *unfocused*, keeping the keyboard on the list. Corollary: once focused **inside** a live terminal, xterm consumes arrows/Enter (cursor sequences) before they bubble to `App`'s window listener; to drop back to list-nav, **click the active live row** (releases focus to the list), click a **non-live** (Recent) row, or arrow-navigate before entering one ‚Äî and to re-enter the terminal, `‚èé` / `‚Üí` / click inside it. (`‚åò[` / `‚åò]` use `e.key`, not `e.code` ‚Äî the Option-key gotcha above is ‚åò‚å•-only.)
- **Preferences modal + New-menu focus:** `SettingsModal` is a paged **Preferences** modal ‚Äî a left nav (**App** / **Shortcuts**) over a scrim ‚Äî that mirrors the app's surfaces (head + nav `--paper-sunken` = the chrome / unified title-bar + rail; page `--paper-pane` = the main content) and holds a **fixed height** (sized to the taller Shortcuts page) so it doesn't resize between pages; dismiss via scrim-click / ‚úï / `Esc`. It opens to **App** from the title-bar **gear** or `‚åò,` (the macOS Preferences key), and to **Shortcuts** from the footer `?` button in `TallyRail` or `‚åò?` (bound Shift-agnostically ‚Äî `‚åò/` *or* `‚åò‚áß/`). While open, `App`'s window key handler short-circuits ‚Äî `Esc` closes; `‚åò,` / `‚åò?` toggle (or close) their page ‚Äî so nothing navigates behind the scrim. The **App** page holds one setting ‚Äî a **default directory for new conversations** (`useNewConvoDefault`): a folder + an independent **enabled** toggle (so it can be turned off without losing the path; the toggle's on-fill is **ink, not cobalt** ‚Äî two-accent invariant). When enabled, the rail-head **"+"** and `‚åòN` skip the chooser and start straight in that folder (a since-deleted folder rejects in `ipc.ts` via an `existsSync` guard ‚Üí the renderer falls back to the chooser); the picker reuses `window.api.pickDirectory()`. - **Find-in-conversation + focus-routed `‚åòF`:** `‚åòF` is focus-sensitive. `App.tsx`'s key handler computes `inMain = paneRef.current?.contains(document.activeElement)` (paneRef is on `.sb-pane`); when focus is in the main pane it opens the in-conversation find bar, otherwise it falls through to the rail's cross-conversation fuzzy search (`fuzzy.ts`, unchanged). **That same `inMain` test also gates the Arrow/Enter list-nav** ‚Äî so once you click into the transcript or terminal, the main pane owns the keyboard (the transcript is `tabIndex={-1}` so a click focuses it; arrows then scroll it instead of moving the list selection). The find bar (`TranscriptSearch`) renders **inline in the pane header**, with a magnifier in the header action cluster (left of the pin) that toggles it. Pressing `‚åòF` while the bar is already open (e.g. after clicking into the transcript) **re-focuses** its input ‚Äî `App` bumps a `findFocusReq` counter threaded App‚ÜíMainPane‚ÜíPaneHeader‚ÜíTranscriptSearch, and `TranscriptSearch` focuses + selects on each change (the input otherwise only auto-focuses on mount, so a second `‚åòF` would be a no-op). Find state lives in `MainPane` (`findQuery` / `activeIndex` / `matchCount`; the query is `useDeferredValue`-debounced). The engine is **`useTranscriptSearch`** (in `TranscriptView`): a `TreeWalker` over `.transcript-content` builds a `Range` for each case-insensitive substring (the pure matcher `findMatches` is unit-tested), painted via the **CSS Custom Highlight API** (`::highlight(sb-find)` / `sb-find-current`, `transcript.css`) ‚Äî **no DOM mutation**, so it doesn't fight react-markdown or defeat `MessageBlock`'s memo (the snappiness-first choice). Coverage now **includes collapsed sections**: matches inside closed tool `<details>` and the clamped tail of long results are found + counted, and navigating to one **reveals** it first ‚Äî `revealContainer` opens the native `<details>` directly and asks a clamped result to expand via an `sb-reveal` event (then defers the scroll one frame for the relayout). Only transcript chrome (footer, loading skeleton) is excluded; matches are still found within a single text node. Opening find over a live **Terminal** switches to Formatted on the first keystroke and restores the prior view on close. The rail's live-session count comes from an **all-sessions** tally in App (`liveTally` over `ptys.active`), never the search-filtered rail set ‚Äî so filtering the list never makes the count drop.
- **Tooltips (`data-tip` + `TooltipLayer`):** app-wide hover labels are a single fixed-positioned element (`TooltipLayer`, mounted at the app root) driven by `data-tip` attributes via document-level event delegation ‚Äî it replaced native `title` everywhere (native `title` lags ~1s and resets on the slightest pointer move, so it rarely showed). Fixed positioning escapes scroll/overflow containers (the rail list, the modal's `overflow:hidden`), flips above when near the bottom edge, and clamps to the viewport at the sides. Look is `.sb-tip` in `global.css` (ink-on-paper ‚Äî no accent). Add a tooltip by putting `data-tip="‚Ä¶"` on any element; icon-only buttons keep an `aria-label` for the accessible name. (The only remaining `title=` in the renderer are two React *props* passing the conversation-title string, not HTML attributes.)
- **Theming (`useTheme` ‚Üí `theme.ts` + `themeDom.ts`):** the app theme is `useTheme` (owned once in `App`; `localStorage` `switchboard.theme` = `system`|`light`|`dark`, default **system**). `theme.ts` is the **pure** core ‚Äî `ThemeMode` / `ResolvedTheme` / `resolveTheme` ‚Äî so the test suite imports it under the **node** tsconfig (which has no DOM lib); the browser-side helpers (`readThemeMode`, `systemPrefersDark`, `applyTheme`) live in `themeDom.ts` (renderer-only). Keep that split ‚Äî putting a `window`/`document` reference in `theme.ts` breaks `typecheck:node`, since `test/` pulls test-imported renderer libs into the node project. `applyTheme` writes `data-theme` on `<html>` and reads the **computed** `--paper` to sync the window `backgroundColor` (drift-proof against token retunes). `main.tsx` calls `applyTheme(resolveTheme(readThemeMode(), systemPrefersDark()))` **before `root.render`** ‚Äî the window is hidden until first paint, so this kills the FOUC without a CSP-blocked inline `<head>` script. `system` tracks `matchMedia('(prefers-color-scheme: dark)')` live. The resolved theme threads `App ‚Üí MainPane ‚Üí TerminalDeck ‚Üí TerminalView` (same path as `focusReq`); `TerminalView` holds two xterm theme objects (`LIGHT_THEME` / `DARK_THEME`, a neutral 16-color ANSI palette + cobalt selection) and swaps `term.options.theme` live on change, so a running `claude` recolors in place (the construction effect captures the initial theme; a `[theme]` effect owns changes). Controls: a **sun/moon** title-bar toggle (`TitleBar`, flips light‚Üîdark as an explicit choice) and the Preferences **App ‚Üí Appearance** System/Light/Dark segmented control (reuses `.sb-seg`). Window-bg sync rides `IPC.windowSetBackgroundColor` (`shared/types` + preload `setBackgroundColor` + an `ipcMain.on` calling `BrowserWindow.fromWebContents(e.sender)?.setBackgroundColor`); `main/index.ts` seeds the initial `backgroundColor` from `nativeTheme.shouldUseDarkColors` for the hidden pre-paint frame.
