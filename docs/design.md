# Switchboard — Design invariants

> Visual & UX rules that must not be broken — read before any UI / CSS change. Lifted from the repo `CLAUDE.md` (see its **Deep reference** index); normative. Every visual change must be authored and verified in BOTH light and dark themes — and, just the same, must support and be considered for BOTH agents (Claude Code and Codex). The two-theme and two-agent invariants are peers: a change that only works in one theme, or for one agent, is unfinished.

- **Two accent colors, each with one job.** Cobalt (`--live`, `#1F5AE6`) = a session is *alive*; the only *action* that may carry it is the one that brings a conversation to life — the right-click **Resume** (cobalt-on-hover, the live counterpart to Stop's red). A warm red (`--danger`, `#d8392c`) = *negative / destructive* (End/Stop controls, error states). Everything else — pins, selection, metadata — stays ink/graphite. The invariant holds in **new chrome too**: the Updates control in Preferences is strictly neutral (ink / paper / gray) — no cobalt, no red — including the **update-available attention dot** on the title-bar gear + the Application nav row, which is a neutral **ink** dot, not a colored badge (an update is neither *alive* nor *destructive*); and the ⋮ row-menu, sidebar headers, and settings buttons (below) all stay grayscale. **Selection is a single tier** — the main pane always owns the keyboard, so there's no list-focus state: the selected row carries its colour as a thin **1px ring**, **cobalt** for a *live* one and neutral grey/ink for the rest (cobalt stays liveness-only, so a cobalt ring = the selected conversation is live), with **no drop shadow or lift**. Mouse-**hover** keeps its own card lift — the theme-aware `--shadow-sel-hover` token: in **light** a dark drop shadow, in **dark** a neutral **light glow** (a *black* shadow is invisible on the near-black rail) — kept narrow + downward-biased so it never hard-cuts at `.sb-rail-body`'s `overflow` clip or the scrollbar gutter; the selected/hovered row is `z-index:1` so that lift paints over neighbouring live cards. (The old focus-tiered *loud* selection shadow and its `--shadow-sel` token were removed when list-focus went away.)
- **Row controls live in the right gutter.** In the left pane's rows, the liveness indicator and the pin toggle share one right-hand slot.
  - **At rest, a live row shows the cobalt liveness dot** (everything else is blank), whose form encodes the conversation's turn-state: **breathing** = a turn is in progress · **a cobalt ripple** (rings pulsing outward from a briefly-swelling dot) = Claude is blocked on your reply (an AskUserQuestion / ExitPlanMode, unread) · **solid** = the turn finished and you haven't viewed it since (unread) · **hollow ring** = finished and already seen (up to date). The solid/unread state can also be set **by hand** (`⇧⌘U`, **Option+click** on a live row, or the row's ⋮ menu — see the Liveness model in [architecture.md](architecture.md)).
  - **The ⋮ row menu replaces both the pin icon and the old right-click menu.** A hover-revealed ⋮ button in the gutter opens the row's actions (pin/unpin, mark read/unread, Resume, Stop, rename, …) in one place — left-click only; there's no separate context menu. **Menu icons reflect the END state**, not the current one: a row that's *not* pinned shows a **filled** pin labelled "to pin" (the state the click produces), and the read/unread item shows the dot it will leave behind. The menu **fades out on close** (~420ms) rather than vanishing, and **auto-closes after ~1.5s un-hovered** — but **never while the pointer is over it** (the timer only runs once you leave). The menu's **read/unread dot is spec'd identically to the liveness dot** — gray at rest, **cobalt on hover** — so the same affordance reads the same everywhere (and stays inside the two-accent rule).
  - **No inline Stop.** A live session is ended by the **Stop** button in the pane header, or the live row's ⋮ menu → **Stop session**.
  - **Pane header cluster:** keeps its own pin on the **left**; there the **Terminal** view-toggle carries a **static solid cobalt dot** — it marks that the session is live but no longer encodes turn-state. The breathing/ripple liveness animation lives only on the left-pane rows.
  - **Drag-to-reorder (Pinned and Live, always on).** Each section reorders within itself via `useRowReorder` — one instance per section, partitioned by CSS selector (`.sb-row.pinned` vs `.sb-row.live:not(.pinned)`, so a pinned-live row is owned by the Pinned instance). The moving card is a `position:fixed` clone on `<body>` so it floats over the head instead of being clipped by the rail body's overflow; its vertical travel is **clamped to a partial overshoot** past its section (≈ half a row — the card's center can't leave the section), not free. Siblings slide live to open the drop gap (fixed section height); on drop the clone eases into its slot *while* dropping out of the lifted look at the same time, so release reads as one motion, not a slide-then-snap-flat.
- **The pane orders by stable keys, never live activity.** Rows hold position by pin order (**Pinned**) and a **manual order** (**Live** — every newly-live session, new or resumed, lands on top, and you drag to reorder; held in `useLiveOrder`, ephemeral) — never `lastActivity` — so a session emitting output never makes a live row jump (regression fixed in `ce407fa`; the manual order now makes this structural — Live no longer auto-sorts by `startedAt` at all, a row holds its slot until you move it). The **Recent** section is the deliberate exception: a recency history of *not-live* conversations, sorted by `mtime`. Pinning a row jumps it to the top of **Pinned** — but that's a user action, not activity. **The whole pane renders through `ConversationRow`** — the unified left pane *is* the left column now (no separate sidebar): card chrome (`.sb-row.card`) marks **live rows only** (a white card = a live session, so a non-live pinned row renders flat like Recent), and Pinned/Live add the live click-to-jump; the pin-on-hover gutter toggle is a shared `.sb-row` primitive, so it (and the liveness dot) work on Recent's flat rows too. The three sections are individually collapsible, and **Recent** is capped (30) with a *Show more* reveal at the bottom. So `ConversationRow` changes flow into every section. When a row *does* change section — pinned, resumed, or unpinned — it **glides** to its new position via a FLIP animation (`useRailFlip`) instead of teleporting; the old `sb-patch-in` entrance keyframe was retired (it re-fired when a row gained `.card` in place, collapsing then re-expanding a resumed pinned row — the "delete + re-add" flicker), so a row going live just eases its card *background* in. `border-color` and `box-shadow` are deliberately **not** transitioned on `.card` rows, so the selection ring (`.selected`, a single thin 1px tier) is **instant** on select *and* deselect — matching the flat rows, which snap the same way; transitioning them made the ring linger ~120ms (an expo-out tail) when you switched to another conversation. (Card hover snaps too, as a consequence — consistent with the rail's flat rows.)
- **Sidebar section headers (PINNED / LIVE / RECENT) are quiet, not loud.** Each header carries a **1px `--ink-ghost` leader rule** — a lighter-weight line in a **higher-contrast color than the `--rule` grays** (darker than `--rule` on light, lighter on dark — so it gains contrast in *both* themes rather than fading at one end). The section **labels themselves are Hanken** (`--font-ui`), **not** the mono `.label-caps` treatment used elsewhere: normal letter-spacing, size `--text-xs`. Between rows, **lightweight inter-row dividers** separate adjacent rows that are *not* live, *not* selected, and *not* hovered — so the chrome of cards / selection / hover always wins and the dividers only quietly stitch the flat resting rows.
- **Single-family type:** Hanken Grotesk for UI + display (heavy weights carry the display moments); IBM Plex Mono for the terminal + mono metadata. No serif. (One narrow exception: keyboard-shortcut glyphs — `⌘ ⇧ ⌥ ⏎ ↑ ↓` — render in `--font-keys` = the macOS system font, which carries them at consistent size/baseline; Hanken & Plex lack several and fall back per-glyph, which looked small/off-baseline.)
- **Preview ≠ spawn:** selecting a conversation only previews it (read-only, from the JSONL). A live `claude` process is created ONLY by explicit **Resume** or **New**.
- **Formatted view is read-only** (rendered from the JSONL); typing happens in the **Terminal** view. It is also the surface for **find-in-conversation** (`⌘F` when the main pane is focused) — and find now **reveals** collapsed sections (opens a closed tool `<details>`, expands a clamped result) when you navigate to a hidden match; only transcript chrome is excluded. The Formatted view also carries **jump-to-edge pills** (**Jump to top** / **Jump to bottom**) — center-aligned, **Hanken** text plus an arrow glyph, each shown **only when scrolled away from that edge** (top pill hidden at the top, bottom pill hidden at the bottom) and jumping with an **instant snap**, not a smooth scroll. They stay strictly neutral (ink / paper / gray) per the two-accent rule.
- **Transcript rendering is one `MessageBlock` per *group*, not per message.** `lib/messageGroups.ts#buildGroups` coalesces consecutive same-source lines under a single header — Claude → Claude → Result → Result renders as one **Claude** then one **Result** (memoized on the transcript so search re-renders keep `MessageBlock`'s memo). **Attribution is by content, not raw role:** the parser tags each `user` line `userKind` — `human` (typed prompt → "You"), `tool_result` (tool output → a **Result**/**Error** header, never "You"), or `interrupted` (the Esc sentinel → a centered muted note); non-conversational `user` lines (slash-command echo, `isMeta`, task-notifications) **and thinking blocks** are dropped at parse time. A single hairline rule separates a **You** (human) turn from the surrounding Claude / Result / Error turns — drawn ONLY at that boundary, never between same-side turns (`.message.has-divider`, computed in `TranscriptView` as `dividerBefore`). The **You** body renders as plain `white-space:pre-wrap` prose (same ink/size as Claude's text, no box, **not** markdown-parsed — so typed line breaks survive). Tool calls/results are **boxless** — the `⚙ Name` head is itself the click-to-expand disclosure (gear only, no chevron, hover-lightens, `user-select:none`) over the sunken JSON/result block — **except an error result, which keeps its red box**. Each header carries a quiet right-aligned timestamp (`format.ts#clockTime`; full date/time on hover via `data-tip`). Code/mono surfaces sit at the **terminal size** (`--text-xs`). **Copy affordances** are hover-revealed icon buttons (shared `CopyButton`, neutral ink, check-flash on click like `copySessionId`): a **turn** copy immediately left of the timestamp — only on **You / Claude** turns that carry prose, copying the **raw markdown source** of the turn's text blocks (`lib/clipboard.ts#turnMarkdown`, pure + unit-tested). The assistant's text is stored AS markdown, so this preserves headers / bold / inline code / fenced code / pipe tables with the author's own blank-line spacing; non-text blocks (tool calls, tool results, images) are skipped, so the ⚙ gear I/O never lands in the copy. (This replaced an earlier DOM-`innerText`-plus-sentinel scheme — the stored source is simpler and exact.) A **corner** copy on every mono block — markdown code (full text), tool-call JSON (`safeStringify(input)`), tool-result text (the full `text` prop, even when clamped), and markdown tables (the **raw markdown source**, sliced from the original via the hast `node`'s position offsets so cell formatting / alignment survive — same as the turn copy; falls back to `lib/clipboard.ts#rowsToMarkdownTable` DOM reconstruction if positions are missing). Buttons carry no text so they're invisible to find; the per-block ones ride a `position:relative` non-scrolling wrapper so they don't scroll with the block.
- Design tokens live in `src/renderer/styles/tokens.css` — reuse the CSS vars; don't hardcode colors.
- **Two themes, one source of truth — a hardcoded color is a dark-mode bug, and EVERY visual change must be authored + verified in BOTH themes.** `tokens.css` defines a **neutral grayscale** light palette in `:root` and a **neutral near-black** dark palette in `:root[data-theme='dark']` (only color + shadow tokens are overridden; type/spacing/radii/motion/layout are shared). Surface tokens: `--paper-sunken` = **chrome** (the rail AND the title bar — `.sb-titlebar` uses `--paper-sunken`, not `--paper`, so they read as one continuous chrome), `--paper-pane` = **main content** (pane / Formatted / terminal), `--paper-raised` = **raised tiles** (live cards, modals, menus), `--paper` = the field/shell behind them. **Dark deliberately INVERTS the elevation** (see the dark-scope comment): the chrome (`--paper-sunken`) is *lighter* than the content canvas (`--paper-pane`), and tiles (`--paper-raised`) lighter still — the reverse of light mode and of the usual dark convention. The dark surfaces were **lightened** off pure near-black — `--paper-pane`, `--paper-sunken`, and `--paper` all sit a step up, with the **terminal background tracked to match** so the terminal doesn't read as a darker hole punched in the pane. Every surface paints from a token, so dark mode is "free" for any component that obeys this — **but "free" covers hue, not feel.** A token flips the *color* per theme; it does NOT carry perceptual *strength* across the light↔dark luminance gap, so any change with a visual dimension (shadow alpha, opacity, contrast, glow) must be **eyeballed in BOTH themes**, not just authored in one. The rail's card hover-lift shadow is the cautionary tale, with a twist: a *low-alpha* black drop shadow is **invisible** on the near-black rail (no contrast at the dark end), so the theme-aware `--shadow-sel-hover` token shares one geometry but **boosts the alpha to near-opaque in dark** — a faint black vanishes on the `--paper-sunken` rail, while an opaque one darkens toward #000 and reads (light keeps the soft low-alpha black). The cobalt **Resume** button's hover reuses the same token. "I changed a visual and only looked in light" is unfinished work. The **two-accent invariant holds in both themes**: cobalt = alive, red = destructive (both brightened for dark, soft/wash tints dark-tinted); pins/selection/metadata stay ink. **The ONE sanctioned exception to grayscale-plus-two-accents is syntax highlighting** — the Formatted view's fenced code blocks carry a muted, full-color syntax palette (highlight.js token colors, themed per light/dark in `transcript.css` as `.hljs-*`). This is the only place any hue beyond cobalt (liveness) / red (errors) appears, and it appears ONLY inside `.hljs` code — a deliberate, tightly scoped exception, not a loophole; nothing outside fenced code may introduce a third hue. One-off shadows use the `--shadow-color` channel (`rgba(var(--shadow-color), α)`) and overlays use `--scrim`. The Preferences modal mirrors these surfaces (chrome head+nav, content page — see the Preferences pointer in [architecture.md](architecture.md)); its **settings buttons are a light gray `--btn`, never white** — and because the elevation inverts, "lighter" is **not one shared value**: light mode's `--btn` is a hair off white, dark mode's is a readable gray, each picked per-theme rather than reused. The active theme is a JS-resolved `data-theme` on `<html>` (System/Light/Dark), **never** a bare `@media (prefers-color-scheme)` — so explicit Light/Dark beats the OS. Preferences → Appearance also exposes a **dark dock-icon toggle** (Light / Dark) that is **independent of the app theme** — it picks which dock-icon variant macOS shows, and does **not** follow (or change) the Light/Dark UI choice; the two are orthogonal settings. See the **Theming** pointer in [architecture.md](architecture.md).
- **Two agents, one experience — every change must support BOTH Claude Code and Codex, exactly as it must both themes.** Switchboard browses conversations from both agents in one unified list; a feature that only works for one is unfinished, the same way a hardcoded color is a dark-mode bug. Everything keys off `ConversationMeta.agent` (`'claude' | 'codex'`): the list, transcript, liveness dot, **resume / new**, **rename**, and the info modal all branch on it. Claude reads `~/.claude/projects/` JSONL; Codex reads `~/.codex/sessions/` rollouts (plus its `state_*.sqlite` + durable `session_index.jsonl` for titles). Resume types `claude --resume <id>` vs `codex resume <id>`; rename appends Claude's own `custom-title` line vs calls Codex's app-server `thread/name/set`; tokens render **per-agent** categories and are **never unified**. The `AGENTS` registry (`shared/types.ts`) carries each agent's label + logo, surfaced by `AgentLogo` — and **logos are ink (`currentColor`), never an accent**, so they obey the two-accent invariant and track their text. The seam is the **main-process agent split** (`sessions/{parser, codexParser, codexThreadsDb, codexSessionIndex, rename, codexRename}`, the dual-root `indexer`/`watcher`, and a per-agent boot command in `pty/manager`); the renderer consumes a **normalized** transcript/meta contract so rendering stays agent-agnostic. When you add or change anything, ask "does this hold for Codex too?" and verify both — exactly like light/dark. See the **Multi-agent (Codex)** pointer in [architecture.md](architecture.md).
