- **Contract:** `src/shared/types.ts` ‚Äî IPC channel names + data types; no node/DOM imports (both sides import it). `ConversationMeta` also carries `turnState` / `turnEndedAt` / `lastActivityAt` / `awaitingTool` / `sessionKind` (the last is read verbatim from the on-disk field; `sessionKind:"bg"` marks a `/bg` daemon **background-job** session, which the indexer **drops** ‚Äî matching Claude Code hiding those from `/resume` ‚Äî so a backgrounded fork never surfaces as a phantom duplicate of the conversation it forked from); `LiveState` is the renderer's resolved working/asking/awaiting/quiet. `IPC.sessionsRename` + `SwitchboardApi.renameConversation(sessionId, title)` are the **only** renderer‚Üímain write into a conversation (an empty title clears back to the auto-generated one). `TranscriptBlock` has **no `thinking` variant** (dropped at parse time); `TranscriptMessage` carries `userKind` (`human` / `tool_result` / `interrupted`) so the renderer attributes `user` lines without re-deriving it (see the Transcript-rendering invariant in [design.md](design.md)). The **self-update** channels (`updatesGetInfo` / `updatesCheck` / `updatesRun` / `updatesProgress` / `updatesRelaunch`) and `SwitchboardApi.getUpdateInfo` / `checkForUpdates` / `runUpdate` / `onUpdateProgress` / `relaunchForUpdate` are the renderer's window into the updater (`updatesProgress` is the one **main‚Üírenderer push**, streaming the update's log lines; the rest are invoke/handle) ‚Äî see the Self-update bullet. The **dark dock-icon** (`windowSetDockIcon`) and **startup/refresh fade** (`appRefreshStart` / `appRefreshEnd`, both main‚Üírenderer pushes) channels round out the chrome ‚Äî see their bullets below.
- **Main:** `index.ts` (window/lifecycle/smoke, + a dev-only dock icon from `build/icon.png` when unpackaged, + a dev-only window-title suffix from `SWITCHBOARD_DEV_LABEL`), `ipc.ts` (handlers; owns watcher + PtyManager ‚Äî also the `windowSetDockIcon` handler that swaps the dock to `build/icon-dark.png` and the `appRefreshStart`/`appRefreshEnd` pushes that bracket the ‚åòR zoom-wiggle, see their bullets), `updater.ts` + `updater-core.ts` (the self-update engine, see the Self-update bullet), `sessions/{parser,indexer,watcher,rename}.ts`, `pty/manager.ts`, `menu.ts` (custom app menu ‚Äî ‚åòR‚ÜíRefresh, no reload roles; see the ‚åòR gotcha in [gotchas.md](gotchas.md)), `windowState.ts` (persists window bounds across launches), `trafficLights.ts` (keeps the native macOS traffic lights aligned to the **page zoom** ‚Äî the OS buttons don't move when ‚åò+/‚åò‚àí scales the renderer, so they'd drift out of center; the renderer pings `IPC.windowSyncTrafficLights` on every `resize` (which fires on every zoom change) and main repositions via `BrowserWindow.setWindowButtonPosition`, scaled by `webContents.getZoomFactor()` ‚Äî buttons stay centered + proportionally gapped to the wordmark; resting inset is nudged a hair up-and-left of macOS's centered default, and sync is **suppressed across the ‚åòR refresh zoom-nudge** so the lights don't wobble with it). `sessions/rename.ts#appendCustomTitle` is the lone JSONL **write** ‚Äî it appends Claude Code's own `custom-title` line (`fs.appendFile`, O_APPEND, atomic for a small line, safe alongside a live `claude`); the `IPC.sessionsRename` handler appends then **re-indexes + broadcasts `sessionsChanged` immediately** so the new title lands now rather than ~600ms later when the watcher catches the same write. The PtyManager spawns `$SHELL -l -i` then types `claude --resume <id>` / `claude --session-id <id>` (login shell = real PATH; see README "Why a login shell?"). It **gates that boot on the first renderer `resize`**, not just the shell prompt: the PTY spawns at a placeholder 80√ó30, so booting claude before the real terminal size arrives makes it replay the entire resume transcript at 80√ó30, and the real size landing mid-replay corrupts claude's cursor math into **real blank rows** in the buffer (which only a later resize/relayout clears). Gating boot on `shellReady && sized` (with a ~2.5s fallback for a terminal that's never shown) makes claude replay at the true size from its first line. PTY busy/idle (output activity, `CONFIG.busyWindowMs` = 800ms) now **only** gates the LRU cap (`CONFIG.maxLivePtys`, **default 8**, user-configurable in Preferences ‚Üí Application (bounds 2‚Äì14), persisted by `useMaxLiveSessions` ‚Üí pushed to the PtyManager over `IPC.ptySetMaxLive`; evicts idle sessions only, never busy work; reaching the cap raises a blocking center-top **capacity modal**, `CapWarningModal`) ‚Äî it does **not** drive the liveness dot at all (a live TUI repaints constantly ‚Äî every keystroke echoes as output ‚Äî so PTY "busy" is ~always true; the transcript turn-state alone drives the dot).
- **Multi-agent (Codex):** Switchboard is a two-agent browser; everything keys off `ConversationMeta.agent` (`'claude' | 'codex'`), with an `AGENTS` registry (`shared/types.ts`) carrying each agent's label + logo (surfaced by `AgentLogo`, painted in `currentColor` so it stays ink, never accent). **Storage differs:** Claude writes `~/.claude/projects/<dashed-cwd>/<id>.jsonl` (cwd-bucketed; the filename stem IS the id and the `--resume` token); Codex writes `~/.codex/sessions/YYYY/MM/DD/rollout-<iso>-<uuid>.jsonl` (date-bucketed, cwd lives *inside* the file) plus a versioned `state_*.sqlite` index whose `threads.title` holds the (renamable) title. `indexer`/`watcher` scan **both roots** and merge metas **by cwd** ‚Äî a repo's Claude and Codex conversations land in one group ‚Äî filtering Claude `sessionKind:"bg"`, Codex non-interactive rollouts (`originator !== 'codex-tui'`), and archived. **Parse:** `codexParser.ts` walks the rollout (`response_item` / `event_msg`) into the SAME normalized `TranscriptMessage` / `TranscriptBlock` shapes the Claude parser emits, so the renderer stays agent-agnostic; meta (preview / cwd / model / tokens / turn-state) comes from the rollout, and the displayed title is resolved by `resolveCodexTitle` (`codexSessionIndex.ts`) from `codexThreadsDb.readCodexThreads` (read-only `node:sqlite` ‚Üí `threads.title` + `first_user_message`) plus `readCodexSessionNames` (the durable `session_index.jsonl`): a *distinct* `threads.title`, else the newest `session_index.jsonl` name, else the rollout-derived title ‚Äî mirroring Codex's own list (see the write-‚âÝ-read gotcha in [gotchas.md](gotchas.md)). **Spawn:** `pty/manager` picks the boot command per agent ‚Äî `claude --resume <id>` / `claude --session-id <id>` vs `codex resume <id>`; new Claude has its id at spawn, new Codex spawns **provisional** and late-binds to its rollout (`matchProvisionalCodex` ‚Üí `pty:bound`; see gotchas). **Rename:** `ipc.ts` `sessionsRename` dispatches by agent ‚Äî Claude appends its own `custom-title` JSONL line (`rename.ts`), Codex calls the app-server `thread/name/set` over NDJSON stdio (`codexRename.ts`) ‚Äî then re-indexes immediately. **New-conversation** adds an agent axis: `useAgentAvailability` (a cached login-shell `command -v claude; command -v codex` probe) + `useNewConvoDefaultAgent` (None / Claude / Codex pref) drive `NewConversationMenu`'s segmented control, which auto-collapses to a single agent when only one is installed. **Tokens** render per-agent categories and are **never unified** (Claude cache tiers vs Codex input / cached / output / reasoning + a context gauge). The **two-agent invariant** (every change must support both, the peer of the two-theme rule) lives in [design.md](design.md); the agent traps (`cleanAgentEnv`, new-Codex late-bind, rename write-‚âÝ-read, the `node:sqlite` / availability notes) are in [gotchas.md](gotchas.md). **Known limitation:** `SessionWatcher` globs only `*.jsonl`, so a Codex change touching *only* the state DB ‚Äî e.g. a rename done in the Codex TUI or another process ‚Äî doesn't trigger a re-index until the next jsonl write, the live poll (while a session is active), or restart (our own renames force one, so they're immediate). Watching `state_*.sqlite*` was deliberately avoided: the WAL/SHM sidecars churn on ordinary activity (and `-shm` even on reads) so it would fire constantly ‚Üí frequent full reindexes, and an always-on DB poll would undo the gated live-poll design. A focus-triggered reindex is the cheap fix if this ever matters.
- **Conversation rename + info modal ‚Äî two entry points, one surface (`ConversationInfoModal`).**
  - **Entry points.** The pane title is a button (`PaneHeader`, `.sb-pane-title-btn` ‚Äî darkened-bg hover, no tooltip) ‚Üí opens the modal in **view**; a row's **‚ãÆ menu ‚Üí Session details‚Ä¶** (`TallyRail`'s per-row menu, **below** read/unread) also opens it in **view**. (The `startInEdit` path remains but no entry point uses it now ‚Äî both open ready to read, and the title is editable in place.) App owns `infoModal: {sessionId, edit}` and renders the modal (resolving the target's meta + pty, mirroring the rail's synth-meta fallback); while it's open App's key handler **short-circuits** (Esc closes, like Preferences / the capacity modal).
  - **Title = heading, edited in place.** An always-editable input styled as the heading (`.sb-info-title-input`): commit on **blur**, **Enter** commits *and* closes the modal, **Esc** reverts (a `suppressRef` skips the ensuing blur-commit; `stopPropagation` so App's Esc doesn't also close ‚Äî first Esc backs out, second closes). Commit writes only on an actual change (never pins an auto-title). A `focusedRef`-guarded `[title]` effect re-seeds the draft from incoming title updates (a commit's re-index, or an external `/rename`) only while the user isn't editing. There's deliberately **no** "reset to auto title" affordance ‚Äî it was removed; to drop a custom title, rename it.
  - **Detail sections** ‚Äî caps-headed (`Section` ‚Üí `.sb-info-section` + a `.label-caps` header over a `.sb-info-grid`): **Workspace** (Folder ¬∑ Branch ¬∑ Model) ¬∑ **Activity** (Messages ¬∑ Size ¬∑ Duration) ¬∑ **Tokens** (gated on having any ‚Äî **Context** first [current window, a live snapshot], then the cumulative per-tier totals **Input ¬∑ Output ¬∑ Cache write ¬∑ Cache read** by Anthropic pricing tier, all visible) ¬∑ **Session** (live **Status** [the lone accent: a static cobalt dot] / **Started** first, then **Last active** ¬∑ **Session ID** last).
  - **Font split.** The **data** (row labels + values) is the **terminal mono face** (`--font-mono` on `.sb-modal-info`) at **one size** (`.sb-info-value` = `--text-xs`, so e.g. Session ID and Started read identically; the `mono` Row prop now only drives truncation, not size), while the **chrome** ‚Äî the editable title and the section headers (`.sb-info-section-label`) ‚Äî stays in the **UI face** (Hanken, `--font-ui`/`--font-display`). A deliberate per-dialog split: the readout reads like the terminal it describes, framed by UI-font headers.
  - **Selection + copy.** Values are **selectable** (`.sb-info-value` re-enables `user-select` against the global body `none`) ‚Äî highlight to copy ‚Äî and **only Session ID** carries a one-click **`CopyButton`** (shared, extracted from `MessageBlock`), now the only place to copy the session ID since the right-click menu's **Copy session ID** item was removed. Timestamps (Last active / Started) show the **absolute** time only; token values are **metric-prefixed** (`format.ts#formatMetric`, e.g. `324K` / `18.4M`), and each token row's **key** (not its value) carries a plain description tooltip via `Row`'s `labelTip` (‚Üí `data-tip` on a span hugging the label, e.g. "Base input tokens") ‚Äî no raw counts surfaced.
  - **Where the values come from.** `ConversationMeta` fields computed in `extractMeta`'s existing per-line loop (negligible cost): `sizeBytes` ¬∑ `model` ¬∑ `firstActivityAt`, plus the token set ‚Äî `inputBaseTokens` ¬∑ `cacheWriteTokens` ¬∑ `cacheReadTokens` ¬∑ `outputTokens` (cumulative, **deduped by `message.id`** ‚Äî the same response is written across multiple JSONL lines all repeating identical `usage`, so a naive sum inflates ~3√ó; `inputTokens` is their input subtotal) and `contextTokens` (the **last main-chain turn's** input+cache = current context-window occupancy, the snapshot Claude Code's status line shows ‚Äî sidechain turns are counted in the cumulative sums but never set `contextTokens`). The 5m/1h cache-write split exists in the JSONL (`usage.cache_creation.ephemeral_*`) but is folded into `cacheWriteTokens` here.
  - **Rename flow.** `onRename` ‚Üí `App.renameConversation` ‚Üí `window.api.renameConversation` ‚Üí the main append + immediate re-index, so the new title lands in the rail, the pane header, and the still-open modal together.
- **Liveness model ‚Äî the dot is derived from the transcript turn-state, never terminal output.**
  - **Turn-state extraction.** `sessions/parser.ts#extractTurnState` scans the transcript's main-chain tail into `ConversationMeta.turnState` + `turnEndedAt` + `lastActivityAt` + `awaitingTool`. An assistant `stop_reason` / a `turn_duration` system line ends a turn; a streaming fragment (no `stop_reason` yet) ‚áí in-progress; a trailing **`AskUserQuestion` / `ExitPlanMode`** tool_use ‚áí `awaiting_input` (+ `awaitingTool` ‚Äî these block for the user's reply and are always a lone terminal tool_use). Sidechain lines are ignored.
  - **Non-conversational trailing `user` lines are skipped** (`classifyUserLine`): slash-command echo/output (`/model`), `!` bash I/O, background-task notifications, and `isMeta` caveats are NOT a turn. So running `/model` after a finished turn stays `awaiting`, not falsely `in_progress` ‚Äî only a real typed prompt or a `tool_result` keeps a turn `in_progress`. An **interrupt sentinel** (`[Request interrupted by user]`, written when you press Esc) ends the turn ‚áí `awaiting`. The **`/compact` summary** (`isCompactSummary`) ‚Äî machine-injected continuation context that *replaces* history, not a typed prompt ‚Äî is likewise modeled as a finished turn ‚áí `awaiting`, so a manual `/compact` (after which Claude idles at the prompt) doesn't read as `in_progress` and breathe forever; an auto-compact's continuation assistant line that follows flips it back to `in_progress`. The summary is also dropped from the Formatted view (like the `/compact` echo beside it). *(Claude-only: Codex derives turn-state from task-lifecycle events, not the last line, so it doesn't hit this.)*
  - **Four-state dot.** `resolveLiveState` (extracted to `renderer/lib/liveness.ts`, pure + unit-tested) combines the turn-state with a per-session **seen** marker (`useSeen`, localStorage; advanced when a conversation is selected *and* the window is focused ‚Äî `useWindowFocus`): **working** (in-progress) / **asking** (blocked on your reply ‚Äî AskUserQuestion / ExitPlanMode ‚Äî unread; clears when looked at, like awaiting) / **awaiting** (finished, unread) / **quiet** (finished+seen, or live-but-not-yet-started ‚Äî a session with no turn-state yet is quiet, never working).
  - **Manual read/unread.** `useSeen` also holds an `unread` override map (sessionId ‚Üí markedAt); `resolveLiveState`'s `manualUnread` arg forces **`awaiting`** (the solid dot) for a finished conversation ‚Äî or **`asking`** (the pulse) for one blocked on your reply (`awaiting_input`), so marking a question-state unread returns it to pulsing ‚Äî beating both `lookingNow` and the seen timestamp. Set it three ways: **`‚áß‚åòU`** (toggles the selected conversation); **Option+click** ‚Äî on a live row in the left pane, or anywhere in its live terminal ‚Äî which **always marks unread, never toggles**; or a row's **‚ãÆ menu**. It **auto-expires** once a newer turn lands (`turnEndedAt > markedAt`, derived ‚Äî no mutation), and is cleared by `markRead`: an explicit toggle, **selecting** the conversation (the seen-effect calls `markRead`, not just `markSeen`), or **engaging** it (a click or keystroke in the pane body, via `MainPane`'s capture-phase listener; scrolling doesn't count).
    - *Option+click wiring:* `ConversationRow`'s `onClick` branches on `e.altKey` before its live/non-live dispatch and returns without navigating; the terminal twin is a dedicated capture-phase `mousedown` listener on the `TerminalView` host (`e.altKey && e.button === 0` ‚Üí `markUnread`) whose `stopPropagation` is **load-bearing** ‚Äî it beats `MainPane`'s bubble-phase engage‚Üí`markRead` *and* xterm's own mouse handling, and `manualUnread` beating `lookingNow` is what makes the dot go solid even while the conversation is open+focused.
    - *‚ãÆ row menu* (`TallyRail`, the per-row vertical 3-dot ‚Äî **replaced the old right-click context menu**) offers **Pin/Unpin** and **Session details‚Ä¶** (on every row, live or not), the read/unread item, plus ‚Äî on a live row only ‚Äî **Stop session** (kills the session), and ‚Äî on a not-live row only ‚Äî **Resume** (spawns + focuses via `App.resumeSession`‚Üí`resume`). It opens on **click-to-toggle** *or* **right-click-at-cursor**, and is dismissed by an outside click, **Esc**, or a **rail scroll** (the scroll listener is scoped to the **rail body**, not document-wide, so scrolling the transcript doesn't close it).
  - **Timestamps.** Rows + header show `relTime(lastActivityAt ?? mtime)` (mtime is bumped by resume/metadata writes, so it isn't true last activity).
  - **Carryover demotion.** `resolveLiveState` demotes an `in_progress` turn whose `lastActivityAt` predates the live process's `PtyState.startedAt`: it belongs to a prior, now-dead run ‚Äî you quit/End the app mid-turn (which writes *no* sentinel, unlike Esc), then **resume**, and the new process just replays history and idles at the prompt (without this it would breathe forever). Anything the current process actually does is stamped after `startedAt`, so genuine work ‚Äî including a long-running tool ‚Äî is never demoted.
  - **Known gaps** (nothing on disk distinguishes them): a turn **blocked on a permission prompt**, and `claude` **crashing while its wrapping shell survives** with no resume (there the turn postdates `startedAt`) ‚Äî both still read as *working*.

- **Self-update (`main/updater.ts` + `updater-core.ts`):** an in-app "check + update" wired through the Preferences **Application ‚Üí Updates** setting (`UpdatesSetting`, first in that page). The split is deliberate: `updater.ts` is the **electron-backed** half (`buildInfo()` reads the baked commit + version, `checkForUpdates()`, `runUpdate(onProgress)` streaming log lines, `relaunchForUpdate()`), while `updater-core.ts` is **pure + unit-tested** (`findRepoRootFrom`, `interpretRemoteSha`, `parseFakeUpdate`) so the test suite imports it under the node tsconfig. The build's commit is **baked at build time** via an `electron.vite.config.ts` `define: { __GIT_SHA__ }` (`git rev-parse HEAD`, `'dev'` fallback when not a build). **CHECK** resolves `main`'s tip SHA via **`git ls-remote` over HTTPS in a login shell** and compares it to the baked commit ‚Äî **not** the GitHub REST API (unauthenticated, 60/hr-per-IP, drained behind a shared corporate NAT) and **not** `node:https` (BoringSSL fails under corporate TLS inspection); equal ‚áí up to date, different ‚áí update available ‚Äî binary, since `ls-remote` carries no commit count (see `docs/gotchas.md`); **UPDATE** shells out in a **login shell** (`git pull --ff-only https://github.com/adambgordon/switchboard.git main && npm run setup`), streams output back over `IPC.updatesProgress`, then `relaunchForUpdate()`. The lifecycle (`info` / `check` / `checking` / `phase` / `log` + `runCheck`/`runUpdate`/`relaunch`) lives in the **App-owned `useUpdates` hook**, not in the modal ‚Äî so the check runs **at launch** (the attention dot can appear before Preferences is ever opened) and the "downloaded, awaiting relaunch" state survives the modal closing. `UpdatesSetting` is now a thin presenter over that hook and renders a **single button per state** (a disabled **Checking‚Ä¶** while in flight ‚Üí **Check again** when up-to-date / **Download update** when behind ‚Üí **Relaunch** when done; re-checking keeps the last status text so the version/status block never shifts). The streamed log ("mini terminal") **auto-closes on a successful update** (it stays only while updating or after a failure, where the output is needed). `useUpdates.needsAttention` (an update is available, or one was downloaded but not relaunched) drives a **neutral ink attention dot** on the title-bar **gear** (`TitleBar`) and the **Application** nav row (`SettingsModal`) ‚Äî neutral per the two-accent invariant (see [design.md](design.md)). It mirrors the README **Update Switchboard** flow (`git pull` + `npm run setup`), just driven from inside the app.
- **Renderer state (hooks in `src/renderer/lib/`):** `useSessions` (live index), `usePtys` (live processes), `usePins` (localStorage; pinned session IDs as a **manually-ordered array** ‚Äî `toggle` prepends a new pin / removes, `reorder(from,to)` moves one, and a derived `Set` is exposed for the rail's `.has()` filters; drag-to-reorder commits via `reorder`, pure `reorderArray` unit-tested), `useLiveOrder` (**ephemeral** in-memory ‚Äî the Live section's manual order; like `usePins` but not persisted, since live PTYs don't outlive the app: every newly-live session, new or resumed, lands on top, drag commits via `reorder` reusing `reorderArray`), `useSeen` (localStorage; per-session "seen" timestamps that split the live dot into unread vs seen, plus an `unread` override map with `markUnread`/`markRead` for manual read/unread), `useWindowFocus` (window focus, which gates when "seen" advances), `useLayout` (localStorage; single-pane shape ‚Äî pane width + collapsed, and per-section collapse for pinned/live/recent), `useNewConvoDefault` (localStorage; the default-directory-for-new-conversations preference ‚Äî a folder path + an independent enabled toggle, read by the `+`/‚åòN handlers and the Preferences App page), `useMaxLiveSessions` (localStorage; the max-live-sessions cap ‚Äî default 8, clamped to bounds 2‚Äì14; pushed to the main-process PtyManager via `IPC.ptySetMaxLive` and drives the capacity modal's threshold), `useDarkIcon` (localStorage; - **Transcript render-windowing ‚Üí reverse-infinite-scroll (`TranscriptView.tsx`):** the Formatted view mounts only the **last ~60 message groups** for an instant first paint, then loads older groups **`GROW_STEP` at a time on scroll-up** (lazily, not eagerly). - **Navigation + focus model ‚Äî the main pane always owns the keyboard.** There's no list-focus state: the rail never holds the keyboard, and the selected row's emphasis is a single tier (see the two-accent invariant in [design.md](design.md)). - **Find-in-conversation + focus-routed `‚åòF`:** `‚åòF` is focus-sensitive. `App.tsx`'s key handler computes `inMain = paneRef.current?.contains(document.activeElement)` (paneRef is on `.sb-pane`); when focus is in the main pane it opens the in-conversation find bar, otherwise it falls through to the rail's cross-conversation fuzzy search (`fuzzy.ts`, unchanged) ‚Äî whose field is boxed like the rename title input (`.sb-rail-search-box`/`-input`; the box persists the whole time search is open, not just on focus), with the rail magnifier shown only while closed and, once open, replaced by an inset ‚úï (`.sb-rail-search-clear`) at the field's right edge. **That same `inMain` test also gates the Arrow/Enter list-nav** ‚Äî so once you click into the transcript or terminal, the main pane owns the keyboard (the transcript is `tabIndex={-1}` so a click focuses it; arrows then scroll it instead of moving the list selection).
