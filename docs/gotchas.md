# Switchboard — Gotchas

> Subsystem traps that have bitten before — read before touching the terminal, the menu, fonts, or the build. Lifted from the repo `CLAUDE.md` to keep it lean (see the **Deep reference** index there); these rules are normative.

- **node-pty rebuild:** use `npm run rebuild` (= `node scripts/rebuild-native.mjs`). The `@electron/rebuild` CLI crashes under Node 26 (yargs/ESM), so we call its programmatic API directly.
- **⌘R refreshes the terminal — it does NOT reload (the app installs a custom menu).**
  - **Why reload is destructive here.** Electron's default menu binds ⌘R / ⇧⌘R to the `reload` / `forceReload` roles. A renderer reload destroys every xterm buffer, and the main process keeps **no** PTY output to replay (`PtyManager` only streams live bytes; the renderer-side `ptyStream` backlog is wiped on reload) — so the remounted `TerminalView`s attach to an **empty** backlog and an idle `claude` emits nothing ⇒ every live terminal goes **blank**, recoverable only by a *real dimension change* (which is why a manual zoom was the one thing that "fixed" it).
  - **The custom menu.** `src/main/menu.ts` (`installAppMenu`, called from `index.ts` `whenReady`) replaces the default with mostly **role-based submenus** (`appMenu` / `editMenu` / `windowMenu`) plus a custom **File** (Close Window + Quit) — so the standard menus and shortcuts (copy/paste/quit/close/zoom/window) are untouched. **Don't drop the File menu** — its absence reads as a broken app. (Quit sits in both the app menu, per macOS convention, and File; the shared ⌘Q is harmless since both quit.)
  - **The ⌘R rebind.** A custom **View** drops the reload roles and rebinds **⌘R / ⇧⌘R → a zoom-wiggle refresh** (`refreshFocused`, surfaced as the **Refresh** menu item): nudging `webContents.setZoomLevel` by a fraction of a level (`REFRESH_ZOOM_NUDGE`, ≈0.5) and restoring it relayouts the renderer → the terminal host's `ResizeObserver` → `fitAndResize` → a real SIGWINCH (claude full redraw) **and** a full-page recomposite (which clears a *stale WebGL frame* — a bare terminal resize can't; this is why the refresh is a genuine zoom, not a targeted `term.refresh()`). The nudge is kept **small to minimize the flicker** but must stay large enough to cross a terminal cell-row boundary at any window size ≥ the app minimum — too small and it rounds to the same cols/rows ⇒ no SIGWINCH ⇒ no content redraw; it's restored after ~60ms (an out-and-back within one frame coalesces to a no-op).
  - **Don't reintroduce the `reload` / `forceReload` roles**, and if you add menu items keep building from the role-based template so the standard shortcuts survive.
- **Packaging:** if you have no code-signing identity, package with `CSC_IDENTITY_AUTO_DISCOVERY=false`. macOS may Gatekeeper-prompt on first open of a fresh build.
- **localStorage is per-origin:** dev (`http://localhost`) and the packaged `.app` (`file://`) don't share pins/layout. Expected, not a bug.
- **macOS Option key:** for `⌘⌥` shortcuts use `e.code` (e.g. `'KeyB'`), not `e.key` (Option+B yields `∫`).
- **electron-vite dev does NOT hot-reload the main / preload process — only the renderer.** Editing anything under `src/main/**` or `src/preload/**` and waiting for an auto-restart does nothing; you must **stop and relaunch `npm run dev`**. (This cost real debugging time twice — a change looked inert purely because the running main was stale.) `src/renderer/**` (components, CSS) DOES hot-reload via HMR, so visual/renderer iteration is live.
- **The boot smoke check can hang on EXIT, and a hung instance stalls a running app.** `SWITCHBOARD_SMOKE=1 node_modules/.bin/electron .` prints `SMOKE PASS | pty:true | window:loaded` (it boots fine), but the Electron process can then fail to quit and linger — and a hung smoke instance has stalled a concurrently-running `npm run dev` instance into unresponsiveness (manual force-quit required). Root cause unconfirmed — suspected multi-Electron-instance / teardown contention, **not** a boot regression (it printed PASS). So: **don't run the smoke while a dev or packaged instance is running**, and skip it entirely for renderer/CSS/doc-only changes (HMR + typecheck/build/test cover those). If it hangs, force-quit and confirm no orphaned helpers remain with `ps -axo pid=,command= | grep -i electron` — list by **argv**; `ps -E` env-matching with `-o command=` silently surfaces nothing on this macOS.
- **Spawned agents get a scrubbed env (`cleanAgentEnv` in `pty/manager.ts`) — without it, a nested `claude` writes no transcript.** When Switchboard's dev build is launched from inside a Claude Code session, its `process.env` carries `CLAUDE_CODE_CHILD_SESSION=1` (+ `CLAUDECODE` / `CLAUDE_EFFORT` / `ANTHROPIC_*`); spread into a spawned `claude`, the child runs as a nested session that **suppresses interactive transcript persistence** → no JSONL → invisible to the index → the liveness dot never lights. `cleanAgentEnv()` strips those markers before the PTY spawn (the login shell then re-sources the real profile). No-op for the packaged app (clean Finder env); harmless for Codex. Don't pass `...process.env` raw to the agent spawn.
- **A new Codex rollout's FILE doesn't exist until the first turn — new-Codex correlation is late-bind, not a spawn-time poll.** Codex mints its id lazily and only writes `~/.codex/sessions/.../rollout-*.jsonl` when the first prompt lands (file birthtime = first-turn time, tens of seconds after the logical start); `-c key=nonce` overrides are accepted but never written to disk, so there's no exact correlation key. So the new-Codex PTY spawns **provisional** (placeholder id) and `matchProvisionalCodex` (pure, `codexParser.ts`) binds it to the freshly-indexed rollout on the next re-index — by cwd + first-activity ≥ spawn time, FIFO, excluding live ids — emitting `pty:bound` so the renderer rekeys the row in place. Don't reintroduce a time-boxed poll-from-spawn; the file isn't there yet.
- **Codex rename is write-≠-read: the title lives in a DB, not the rollout.** `codexRename.ts` sends the app-server `thread/name/set` over NDJSON stdio, which writes `state_*.sqlite` `threads.title` — **not** the rollout. The rename only surfaces because the indexer also reads it: `codexThreadsDb.readCodexTitles` opens the newest `state_*.sqlite` via `node:sqlite` **read-only** and overlays that title over the rollout-derived one. `node:sqlite` ships in Electron 42's Node 24 (unflagged), and read-only **honors the WAL** (a fresh rename sits in `-wal` until checkpoint — verified). If you touch the title path, keep write and read in sync or the rename flashes then reverts.
- **Terminal image input rides on bracketed paste.** claude attaches an image on a *paste* — it reads the system clipboard, or parses an image path inside the pasted text — never on plain typed text. So `TerminalView` delivers via the bracketed-paste sequence (`ESC[200~ … ESC[201~`): **Ctrl+V** sends an *empty* one (claude then reads the clipboard image → `[Image #N]`); **drag-drop** wraps the dropped, shell-escaped path(s) in one. The Cmd+V/Ctrl+V roles are **inverted vs. a native terminal**: under xterm, **Cmd+V** is the real (browser) `paste` event while **Ctrl+V** would only send `\x16`. So we own **Ctrl+V** via `attachCustomKeyEventHandler`, and intercept **Cmd+V** in *capture phase* to keep it text-only (an image-only clipboard is suppressed, so claude can't grab it). macOS-only (Cmd+V fires a DOM `paste` event; Ctrl+V doesn't).
- **Terminal renders via the WebGL addon (canvas is the fallback), never the DOM renderer.** `TerminalView` loads `@xterm/addon-webgl` *after* `term.open()` (required ordering); on GPU-context loss (`onContextLoss`) — or if WebGL can't initialize — it swaps to `@xterm/addon-canvas`. WebGL is the faster, higher-fidelity renderer (it draws every glyph each frame *without* clipping it to the cell, which canvas does — that clipping was the source of the box-glyph/whitespace artifacts) and is where xterm.js + VS Code are consolidating. Both honor `customGlyphs` (default on), so box-drawing/block glyphs paint as continuous vector lines. The fallback is **canvas, not the default DOM renderer**: DOM pulls box-glyphs from the font, which breaks `claude`'s table/box borders at our non-integer `fontSize` (11.5) + `lineHeight` (1.3). Don't switch the *primary* off WebGL or the *fallback* to DOM without eyeballing how tables render. (Text stays IBM Plex Mono either way.)
- **Terminal `lineHeight` is kept near-integer for the canvas fallback.** `11.5 × 1.3 ≈ 15px` makes the cell height ~integer. The WebGL primary doesn't need this, but the **canvas fallback** strands blank rows during rapid live redraws at a fractional cell (claude's tool-call output) — intermittent vertical whitespace gaps that only a repaint (scroll, or ⌘ +/- zoom) clears, never selection (selection is a separate render layer). It was `1.32` (15.18px); WebGL fixed those gaps at the root (no per-cell clipping / repaint desync), but keep `1.3` so the fallback path stays clean. Don't raise `lineHeight` to a value that makes the cell fractional again.
- **Terminal Unicode width is pinned to v11.** `TerminalView` loads `@xterm/addon-unicode11` and sets `term.unicode.activeVersion = '11'` right after `term.open()` (before the first PTY write). xterm's built-in default is the Unicode **V6** width table, which measures emoji-presentation chars (✅ `U+2705`, ❌ `U+274C`, …) as **1 cell** — but `claude` lays out tables / box-art assuming the modern **2-cell** emoji width (as every real terminal does), so the 1-cell mismatch shifted every cell after an emoji and knocked the closing `│` border out of alignment (and the half-cell glyph overdraw flickered under partial canvas repaints on scroll/select). V11 widens those codepoints to 2, matching claude. Requires `allowProposedApi: true` (the `term.unicode` interface is proposed API). Don't remove the addon or fall back to V6 — and don't confuse this with grapheme clustering, a separate, costlier feature we deliberately don't enable (V11 is just a corrected width table; the per-char hot path is unchanged from V6).
- **Terminal cursor is hidden — Switchboard defers to Claude's own caret.** `TerminalView` sets `theme.cursor` transparent (`rgba(0,0,0,0)`) with `cursorBlink: false`. `claude` (like most full-screen TUIs) draws and styles its OWN caret — a reverse-video block — so xterm's hardware cursor would just paint a redundant *second* cursor on the same cell (it showed as a cobalt bar beside Claude's block). Transparent leaves Claude's block as the only caret. `cursorStyle: 'bar'` is kept solely so the invisible cursor never recolors the glyph under it (a `block` cursor would), and this subsumes the old `cursorInactiveStyle: 'none'` too (a transparent cursor is invisible focused or not, so arrow-preview shows no stray box). Trade-off: no visible cursor at a bare shell prompt — fine, since this terminal exists to host a live `claude`. Cursor *style* is therefore not a Preference (deliberately not among the App page's settings) — it's Claude's to draw.
- **The terminal MOUSE cursor is forced to text (I-beam) in CSS — distinct from the hidden caret above.** xterm sets `cursor: crosshair` while Option/Alt is held (its rectangular column-select affordance — renders as the macOS `+`) and `cursor: default` when a TUI turns on mouse reporting; `terminal.css` overrides BOTH (plus the base `.xterm`) to `cursor: text`. The selectors carry enough specificity to beat xterm's own class rules (`.sb-term` prefix + matching the same classes — `.column-select.focus` etc.), so **no `!important`**. We don't use rectangular selection and `claude` owns mouse modes regardless of the pointer *shape*, so an I-beam is right at all times. The **Formatted** view matches it (`.transcript-scroll { cursor: text }`) — interactive children (links, the tool toggle `<summary>`, Show-more / copy `<button>`s) keep `pointer` via their own rules. Don't reintroduce the Option crosshair "to match a real terminal" — it was deliberately removed.
- **A hidden terminal's renderer is paused, so it's force-repainted on show.** xterm's `RenderService` installs an `IntersectionObserver` on the screen element and sets `_isPaused` while it isn't intersecting the viewport — which `display:none` triggers — after which `refreshRows` **drops** every paint request. `TerminalDeck` hides inactive live sessions with `display:none`, so a session that animates while hidden (claude's mid-thought spinner) freezes on its last painted frame when you navigate to it, and stays frozen until a redraw is forced (a manual scroll was the only cue). `TerminalView`'s becomes-visible effect fixes it: it **clears that paused flag directly** (`term._core._renderService._isPaused = false`, reached via private API and guarded with optional chaining) before the fit + `term.refresh(0, rows-1)`. The earlier fix — refreshing on two `requestAnimationFrame`s and hoping the observer had un-paused — was unreliable, because Chromium does **not** consistently re-fire the `IntersectionObserver` on a pure `display:none → block` toggle (it recomputes intersection on scroll/layout), so the refresh got dropped while still paused (symptom: a frozen spinner that only a manual scroll woke). Clearing the flag ourselves guarantees the repaint; the observer later sets it correctly (visible ⇒ false), a no-op. One-time per navigation (no per-frame cost). Don't drop it — and don't "fix" it by taking the deck off `display:none`, which would keep every hidden live terminal repainting (a CPU regression against the LRU/pause bounds).
- **Never fit/resize a hidden (`0×0`) terminal — it pushes a bogus tiny size that strands blank rows.** When a deck item goes `display:none` (you switch to another conversation), its host measures `0×0` *and* its `ResizeObserver` fires — but `@xterm/addon-fit` does **not** bail on a zero-size host: it reads the host's `height`/`width:100%` *computed* style (`parseInt("100%")` → 100px) and proposes a ~`13×5` grid. Pushing that to the PTY makes `claude` reflow its whole TUI into 13×5 and then back to full size when you return, and the reflow strands **real blank rows** in the buffer (the "whitespace after tool calls" gap — same symptom as the 80×30 boot bug, but a *different* cause: a live resize, not boot replay). `TerminalView` funnels every resize through one `fitAndResize` that **skips entirely when `host.clientWidth`/`clientHeight === 0`** and **suppresses no-op resizes** (sends only when `cols`/`rows` actually changed), so switching conversations/views sends *zero* resizes when the size is unchanged. Don't reintroduce an unconditional `window.api.resize(...)` after `fit.fit()`, and don't drop the `0×0` guard — the `ResizeObserver` firing on hide is the trigger. (The guard also subsumes the boot case: a hidden mount no longer sends xterm's 80×24 default, so it can't satisfy the boot-size gate at the wrong size.)
- **The initial terminal fit runs before the PTY backlog, on purpose.** `TerminalView` calls `fitAndResize()` once right after `term.open()` + addons and *before* `attachPty`, so the expensive WebGL resize hits an empty terminal instead of a screenful mid-replay — it removed a ~170ms main-thread stall on resume (trace-verified). The rAF + `ResizeObserver` after it are the fallback (host not yet measurable) and the later-resize path; don't move the first fit back behind the flood. The residual *choppy scroll for the first seconds after resume* is GPU/compositor frame-pacing — not main-thread, not cheaply fixable — so don't re-chase it as a main-thread problem.
- **Fonts are warmed before first paint.** `main.tsx` gates `root.render` on `document.fonts.load(...)` for every bundled weight (Hanken 400/500/600/700/800, IBM Plex Mono 400/500/600), raced against a 1500 ms ceiling. `@fontsource` ships each weight as a separate `font-display: swap` file fetched lazily on first use, so a bold run could paint in a fallback face and then **not** repaint when the real face loaded — Chromium leaves the already-painted glyphs stale until a reflow (selecting or zooming the text forced it). Decoding up front closes that window: text paints once, in the right face (~20 ms on local files, behind the `show:false` window). Don't drop the gate or the intermittent bold-font flash returns — and note `document.fonts.ready` *alone* is a no-op here (nothing is pending before render, so the explicit `.load()` calls are what kick off the decode).
- **Don't clamp `white-space: pre-wrap` text with `-webkit-line-clamp`.** The Formatted view caps a long tool result at **12 rendered lines** via a content-box `max-height` (= 12 line-heights, `1.55em` each) on a **padding-less inner clip** element with `overflow:hidden`; the gap below the cut is the panel's own bottom padding. `-webkit-line-clamp` was tried (twice) and fails on `pre-wrap`: it cuts **mid-line** AND makes `scrollHeight === clientHeight`, which broke the overflow check that gates "Show more". That check is now a `ResizeObserver` on the text (a one-shot read raced layout + font-load and dropped the button).
- **Transcript scroll-pinning releases ONLY on a genuine scroll-up — content growing beneath a fixed position must RE-PIN, not release.** `TranscriptView`'s "stick to bottom" pin is dropped only when the user actually scrolls up (`scrollTop` *decreases*). The subtle trap: a gap can open at the bottom because content grew *beneath* a fixed `scrollTop` (tool-result clamps settling after layout, the window growing) — that leaves `scrollTop` **unchanged**, so reading it as a user gesture and releasing the pin is wrong; it must **re-pin**. Conflating the two stranded long conversations mid-history. So `overflow-anchor: none` stays on the scroll container (manual compensation owns the position, not the browser's anchoring), and **every programmatic scroll write updates `lastTopRef`** so our own writes aren't later misread as the user scrolling. Don't treat "scrollTop didn't reach bottom" as "user scrolled away".
- **The transcript loads older groups on SCROLL-UP, not eagerly — reverse infinite-scroll, no idle-grow.** An earlier version eagerly grew the transcript on idle, prepending chunks while pinned to the bottom and re-pinning after each — which produced a visible **1–2s "shake"** when opening a long conversation (re-pinning runs after layout and can't out-race the paint). Now: render only the last **~60 groups**, and load older `GROW_STEP`-at-a-time **only as the user nears the top**. A content `ResizeObserver` re-pins to the bottom **before paint** (ResizeObserver fires after layout but before paint), so async tool-result settling doesn't drop the view. Search / jump-to-top force a full mount. Don't reintroduce eager idle-grow — the shake comes back.
- **The ⋮ row menu's scroll-to-close listener is scoped to the RAIL body, not `document`.** A document-wide capture-phase `scroll` listener (to dismiss the open ⋮ menu when the list scrolls) *also* caught the transcript pane's scrolls — and opening a large conversation re-pins for ~1–2s, spamming scroll events that **slammed the menu shut the instant it opened**. Scroll events don't bubble, so a listener attached to the rail body fires only for rail scrolls — that scoping is the fix, not an optimization. Don't move it back to `document`.
- **The build commit is baked at PACKAGE time, not read at runtime — the updater compares it to `main`, not to the repo's current HEAD.** `electron.vite.config.ts` injects `define: { __GIT_SHA__ }` from `git rev-parse HEAD` (falling back to `'dev'`), so the running app knows the SHA it was *built* from. The updater compares **that** baked SHA to `main` — deliberately **not** the repo's live HEAD, which can drift ahead if you `git pull` without rebuilding. So an "up to date" verdict means *the running binary* matches `main`, not that the checkout does.
- **The updater never uses SSH, and runs ONLY in the packaged app from a locatable source clone.** `origin` is SSH, but a GUI-spawned `git` can't reliably reach the SSH agent — so the check resolves `main`'s tip via **`git ls-remote` over HTTPS** and the update pulls an explicit `https://…` URL (public repo ⇒ unauthenticated read works). The whole path is gated on `app.isPackaged` and only fires when the source repo is found by walking up from `app.getAppPath()` for a `.git` + a `package.json` whose name is `switchboard` — a `.app` dragged to `/Applications` is detached from its clone, so it correctly falls back to printing the manual update command. `SWITCHBOARD_FAKE_UPDATE=current|behind|unknown` forces a check result for eyeballing the update states in dev (inert in production). Don't switch the remote to SSH or assume the running `.app` can find its source.
- **The update CHECK uses `git ls-remote`, NOT the GitHub REST API — two independent reasons the REST API fails behind a corporate network.** (1) **TLS inspection:** the firewall re-signs `api.github.com` with an internal root CA (Palantir's does), and `node:https` then fails 100% with `SELF_SIGNED_CERT_IN_CHAIN` — Electron's Node is backed by **BoringSSL**, which trusts only its bundled Mozilla CA set and ignores `SSL_CERT_FILE`/`NODE_EXTRA_CA_CERTS` (and a Finder-launched GUI app wouldn't inherit those anyway). Electron's `net` module clears *that* hurdle (Chromium's stack + the OS trust store), but — (2) **shared-IP rate limit:** the unauthenticated REST API is capped at **60 req/hr per IP**, and behind a shared corporate NAT that pool is drained by everyone's ambient anonymous traffic, so the check 403s unpredictably (a public app has no token to raise the cap). So the check resolves `main`'s tip SHA with **`git ls-remote <https> main` in a login shell** and compares it to the baked `__GIT_SHA__`: git's smart-HTTP transport (`github.com`, not `api.github.com`) isn't subject to the REST limit, and the login shell gives git the PATH + corporate TLS config a GUI app lacks (the same path `runUpdate`'s `git pull` already uses). Tradeoff: `ls-remote` yields only the tip SHA, so the result is **binary** ('Update available', not 'N commits behind'). Verify the real check on a **packaged** build (`npm run dev` short-circuits it to `unknown`). Don't reach back for the REST API (rate limit) or `node:https` (TLS), and don't relax cert verification.
- **Syntax-highlighting languages are a curated 13-language subset, and the `code` override MERGES className.** `MessageBlock.tsx` registers rehype-highlight with only ~13 languages to keep the bundle lean; an unknown fence renders **unstyled** (tolerated — no throw). The trap: the custom `code` component override must **merge the incoming `className`**, not replace it, or it strips rehype-highlight's injected `hljs language-*` classes and the highlighting silently vanishes. Don't drop the incoming className when overriding `code`, and don't expand the language set without weighing the bundle cost.
