# Switchboard ‚Äî Gotchas

> Subsystem traps that have bitten before ‚Äî read before touching the terminal, the menu, fonts, or the build. Lifted from the repo `CLAUDE.md` to keep it lean (see the **Deep reference** index there); these rules are normative.

- **node-pty rebuild:** use `npm run rebuild` (= `node scripts/rebuild-native.mjs`). The `@electron/rebuild` CLI crashes under Node 26 (yargs/ESM), so we call its programmatic API directly.
- **‚åòR refreshes the terminal ‚Äî it does NOT reload (the app installs a custom menu).** Electron's default menu binds ‚åòR / ‚áß‚åòR to the `reload` / `forceReload` roles, which is destructive here: a renderer reload destroys every xterm buffer, and the main process keeps **no** PTY output to replay (`PtyManager` only streams live bytes; the renderer-side `ptyStream` backlog is wiped on reload) ‚Äî so the remounted `TerminalView`s attach to an **empty** backlog and an idle `claude` emits nothing ‚áí every live terminal goes **blank**, recoverable only by a *real dimension change* (which is why a manual zoom was the one thing that "fixed" it). `src/main/menu.ts` (`installAppMenu`, called from `index.ts` `whenReady`) replaces the default menu with mostly **role-based submenus** (`appMenu` / `editMenu` / `windowMenu`, plus a custom **File** = Close Window + Quit ‚Äî so the standard menus and shortcuts, copy/paste/quit/close/zoom/window, are untouched; **don't drop the File menu** ‚Äî its absence reads as a broken app. Quit sits in both the app menu (macOS convention) and File; the shared ‚åòQ is harmless since both quit) and a custom **View** that drops the reload roles and rebinds **‚åòR / ‚áß‚åòR ‚Üí a zoom-wiggle refresh** (`refreshFocused`, surfaced as the **Refresh** menu item): nudging `webContents.setZoomLevel` by a fraction of a level (`REFRESH_ZOOM_NUDGE`, ‚âà0.5) and restoring it relayouts the renderer ‚Üí the terminal host's `ResizeObserver` ‚Üí `fitAndResize` ‚Üí a real SIGWINCH (claude full redraw) **and** a full-page recomposite (which clears a *stale WebGL frame* ‚Äî a bare terminal resize can't, see the stale-composite note below; this is why the refresh is a genuine zoom and not a targeted `term.refresh()`). The nudge is kept **small to minimize the flicker** but must stay large enough to cross a terminal cell-row boundary at any window size ‚â• the app minimum ‚Äî too small and it rounds to the same cols/rows ‚áí no SIGWINCH ‚áí no content redraw; restored after ~60ms (an out-and-back within one frame coalesces to a no-op). **Don't reintroduce the `reload` / `forceReload` roles**, and if you add menu items keep building from the role-based template so the standard shortcuts survive.
- **Packaging:** if you have no code-signing identity, package with `CSC_IDENTITY_AUTO_DISCOVERY=false`. macOS may Gatekeeper-prompt on first open of a fresh build.
- **localStorage is per-origin:** dev (`http://localhost`) and the packaged `.app` (`file://`) don't share pins/layout. Expected, not a bug.
- **macOS Option key:** for `‚åò‚å•` shortcuts use `e.code` (e.g. `'KeyB'`), not `e.key` (Option+B yields `‚à´`).
- **Terminal image input rides on bracketed paste.** claude attaches an image on a *paste* ‚Äî it reads the system clipboard, or parses an image path inside the pasted text ‚Äî never on plain typed text. So `TerminalView` delivers via the bracketed-paste sequence (`ESC[200~ ‚Ä¶ ESC[201~`): **Ctrl+V** sends an *empty* one (claude then reads the clipboard image ‚Üí `[Image #N]`); **drag-drop** wraps the dropped, shell-escaped path(s) in one. The Cmd+V/Ctrl+V roles are **inverted vs. a native terminal**: under xterm, **Cmd+V** is the real (browser) `paste` event while **Ctrl+V** would only send `\x16`. So we own **Ctrl+V** via `attachCustomKeyEventHandler`, and intercept **Cmd+V** in *capture phase* to keep it text-only (an image-only clipboard is suppressed, so claude can't grab it). macOS-only (Cmd+V fires a DOM `paste` event; Ctrl+V doesn't).
- **Terminal renders via the WebGL addon (canvas is the fallback), never the DOM renderer.** `TerminalView` loads `@xterm/addon-webgl` *after* `term.open()` (required ordering); on GPU-context loss (`onContextLoss`) ‚Äî or if WebGL can't initialize ‚Äî it swaps to `@xterm/addon-canvas`. WebGL is the faster, higher-fidelity renderer (it draws every glyph each frame *without* clipping it to the cell, which canvas does ‚Äî that clipping was the source of the box-glyph/whitespace artifacts) and is where xterm.js + VS Code are consolidating. Both honor `customGlyphs` (default on), so box-drawing/block glyphs paint as continuous vector lines. The fallback is **canvas, not the default DOM renderer**: DOM pulls box-glyphs from the font, which breaks `claude`'s table/box borders at our non-integer `fontSize` (11.5) + `lineHeight` (1.3). Don't switch the *primary* off WebGL or the *fallback* to DOM without eyeballing how tables render. (Text stays IBM Plex Mono either way.)
- **Terminal `lineHeight` is kept near-integer for the canvas fallback.** `11.5 √ó 1.3 ‚âà 15px` makes the cell height ~integer. The WebGL primary doesn't need this, but the **canvas fallback** strands blank rows during rapid live redraws at a fractional cell (claude's tool-call output) ‚Äî intermittent vertical whitespace gaps that only a repaint (scroll, or ‚åò +/- zoom) clears, never selection (selection is a separate render layer). It was `1.32` (15.18px); WebGL fixed those gaps at the root (no per-cell clipping / repaint desync), but keep `1.3` so the fallback path stays clean. Don't raise `lineHeight` to a value that makes the cell fractional again.
- **Terminal Unicode width is pinned to v11.** `TerminalView` loads `@xterm/addon-unicode11` and sets `term.unicode.activeVersion = '11'` right after `term.open()` (before the first PTY write). xterm's built-in default is the Unicode **V6** width table, which measures emoji-presentation chars (‚úÖ `U+2705`, ‚ùå `U+274C`, ‚Ä¶) as **1 cell** ‚Äî but `claude` lays out tables / box-art assuming the modern **2-cell** emoji width (as every real terminal does), so the 1-cell mismatch shifted every cell after an emoji and knocked the closing `‚îÇ` border out of alignment (and the half-cell glyph overdraw flickered under partial canvas repaints on scroll/select). V11 widens those codepoints to 2, matching claude. Requires `allowProposedApi: true` (the `term.unicode` interface is proposed API). Don't remove the addon or fall back to V6 ‚Äî and don't confuse this with grapheme clustering, a separate, costlier feature we deliberately don't enable (V11 is just a corrected width table; the per-char hot path is unchanged from V6).
- **Terminal cursor is hidden ‚Äî Switchboard defers to Claude's own caret.** `TerminalView` sets `theme.cursor` transparent (`rgba(0,0,0,0)`) with `cursorBlink: false`. `claude` (like most full-screen TUIs) draws and styles its OWN caret ‚Äî a reverse-video block ‚Äî so xterm's hardware cursor would just paint a redundant *second* cursor on the same cell (it showed as a cobalt bar beside Claude's block). Transparent leaves Claude's block as the only caret. `cursorStyle: 'bar'` is kept solely so the invisible cursor never recolors the glyph under it (a `block` cursor would), and this subsumes the old `cursorInactiveStyle: 'none'` too (a transparent cursor is invisible focused or not, so arrow-preview shows no stray box). Trade-off: no visible cursor at a bare shell prompt ‚Äî fine, since this terminal exists to host a live `claude`. Cursor *style* is therefore not a Preference (deliberately not among the App page's settings) ‚Äî it's Claude's to draw.
- **A hidden terminal's renderer is paused, so it's force-repainted on show.** xterm's `RenderService` installs an `IntersectionObserver` on the screen element and sets `_isPaused` while it isn't intersecting the viewport ‚Äî which `display:none` triggers ‚Äî after which `refreshRows` **drops** every paint request. `TerminalDeck` hides inactive live sessions with `display:none`, so a session that animates while hidden (claude's mid-thought spinner) freezes on its last painted frame when you navigate to it, and stays frozen until a redraw is forced (a manual scroll was the only cue). `TerminalView`'s becomes-visible effect fixes it: it **clears that paused flag directly** (`term._core._renderService._isPaused = false`, reached via private API and guarded with optional chaining) before the fit + `term.refresh(0, rows-1)`. The earlier fix ‚Äî refreshing on two `requestAnimationFrame`s and hoping the observer had un-paused ‚Äî was unreliable, because Chromium does **not** consistently re-fire the `IntersectionObserver` on a pure `display:none ‚Üí block` toggle (it recomputes intersection on scroll/layout), so the refresh got dropped while still paused (symptom: a frozen spinner that only a manual scroll woke). Clearing the flag ourselves guarantees the repaint; the observer later sets it correctly (visible ‚áí false), a no-op. One-time per navigation (no per-frame cost). Don't drop it ‚Äî and don't "fix" it by taking the deck off `display:none`, which would keep every hidden live terminal repainting (a CPU regression against the LRU/pause bounds).
- **Never fit/resize a hidden (`0√ó0`) terminal ‚Äî it pushes a bogus tiny size that strands blank rows.** When a deck item goes `display:none` (you switch to another conversation), its host measures `0√ó0` *and* its `ResizeObserver` fires ‚Äî but `@xterm/addon-fit` does **not** bail on a zero-size host: it reads the host's `height`/`width:100%` *computed* style (`parseInt("100%")` ‚Üí 100px) and proposes a ~`13√ó5` grid. Pushing that to the PTY makes `claude` reflow its whole TUI into 13√ó5 and then back to full size when you return, and the reflow strands **real blank rows** in the buffer (the "whitespace after tool calls" gap ‚Äî same symptom as the 80√ó30 boot bug, but a *different* cause: a live resize, not boot replay). `TerminalView` funnels every resize through one `fitAndResize` that **skips entirely when `host.clientWidth`/`clientHeight === 0`** and **suppresses no-op resizes** (sends only when `cols`/`rows` actually changed), so switching conversations/views sends *zero* resizes when the size is unchanged. Don't reintroduce an unconditional `window.api.resize(...)` after `fit.fit()`, and don't drop the `0√ó0` guard ‚Äî the `ResizeObserver` firing on hide is the trigger. (The guard also subsumes the boot case: a hidden mount no longer sends xterm's 80√ó24 default, so it can't satisfy the boot-size gate at the wrong size.)
- **The initial terminal fit runs before the PTY backlog, on purpose.** `TerminalView` calls `fitAndResize()` once right after `term.open()` + addons and *before* `attachPty`, so the expensive WebGL resize hits an empty terminal instead of a screenful mid-replay ‚Äî it removed a ~170ms main-thread stall on resume (trace-verified). The rAF + `ResizeObserver` after it are the fallback (host not yet measurable) and the later-resize path; don't move the first fit back behind the flood. The residual *choppy scroll for the first seconds after resume* is GPU/compositor frame-pacing ‚Äî not main-thread, not cheaply fixable ‚Äî so don't re-chase it as a main-thread problem.
- **Fonts are warmed before first paint.** `main.tsx` gates `root.render` on `document.fonts.load(...)` for every bundled weight (Hanken 400/500/600/700/800, IBM Plex Mono 400/500/600), raced against a 1500 ms ceiling. `@fontsource` ships each weight as a separate `font-display: swap` file fetched lazily on first use, so a bold run could paint in a fallback face and then **not** repaint when the real face loaded ‚Äî Chromium leaves the already-painted glyphs stale until a reflow (selecting or zooming the text forced it). Decoding up front closes that window: text paints once, in the right face (~20 ms on local files, behind the `show:false` window). Don't drop the gate or the intermittent bold-font flash returns ‚Äî and note `document.fonts.ready` *alone* is a no-op here (nothing is pending before render, so the explicit `.load()` calls are what kick off the decode).
- **Don't clamp `white-space: pre-wrap` text with `-webkit-line-clamp`.** The Formatted view caps a long tool result at **12 rendered lines** via a content-box `max-height` (= 12 line-heights, `1.55em` each) on a **padding-less inner clip** element with `overflow:hidden`; the gap below the cut is the panel's own bottom padding. `-webkit-line-clamp` was tried (twice) and fails on `pre-wrap`: it cuts **mid-line** AND makes `scrollHeight === clientHeight`, which broke the overflow check that gates "Show more". That check is now a `ResizeObserver` on the text (a one-shot read raced layout + font-load and dropped the button).
